# v0.09 ✅ Complete

## Specification review and editing ✅ Complete

The specification documents have grown over time and need to be reviewed.

This work will review all documents in `doc/spec/` and bring sections together
so that the specification is cohesive and comprehensive. Another important
aspect to this work will be to validate the specification against the codebase
as it stands - it is important that any gaps between the specification and the
code are resolved by either updating the spec or raising the need to modify the
code to meet the expected functionality.

This work also folds the former KMDB Primer into the spec itself, as the new
[System Overview](../../spec/01_overview.md) front-matter section, rather than
keeping it as a separate document.

**Dependency note:** the [Integration Guide](../0_10.md) work item (moved out to
its own roadmap, `0_10.md`, as a significant body of code and documentation in
its own right) depends on this work as a prerequisite — its guide prose cites
spec section numbers throughout, so this review is what confirms those
sections are correct, complete, and current before the guide is written
against them.

  [plan_0_09_spec_review_and_primer_fold.md](../../plans/completed/plan_0_09_spec_review_and_primer_fold.md)

## Release Checklist ✅ Complete

Releasing KMDB will require a sequence of publishing steps that ensure that each
package is published in the correct order. We'll need a release checklist
template that can be followed for each release.

A `README.md` file will be added to `docs/releasing` that describes the release
process and provides the template checklist.

Each release will have a copy of the template created in `docs/releasing` and
named after the version being released - e.g. `0.1.0-dev.1.md`. The version
provided in the main [`pubspec.yaml`](../../../pubspec.yaml) will be the designated
version being released. It may be that some of the KMDB packages do not need to
be updated in a release so it is reasonable that different packages may have
different minor or patch versions but they must never have different major
versions.

The [Release Checklist specification](../../spec/28_release_checklist.md) needs to
be included for each release as a pre-requisite series of tests that must be
completed prior to a release. Each item in the Release Checklist specification
must have an associated check-list entry that indicates if the test has been
completed (`[x]`) or if it is unnecessary for the release (`[-]`).

## Vault file export ✅ Complete

The CLI should include a `vault export` command that will export a specific
vault file to the path given in the `--output` parameter.

- If `--output` is a file path, the vault file will be exported to that path.
- If `--output` is a directory, the vault file will be placed into the requested
  directory and the file will be named based on the `originalName` property in
  `metadata.json` - if the `originalName` property does not exist, the vault
  file with be exported to the path with a file name of `blob`.

Note that the CLI usage indicates that the user can run `vault help` but this
fails on a database where Vault has not been initialised
(`Error: Vault is not available for this database. Vault storage is initialised automatically when files are first ingested via "vault ingest" or "--import".`).
This needs to be fixed as part of this work if it hasn't already been rectified.

- **Scope:** `kmdb_cli` (`vault_command.dart`, new `vault_export_command.dart`,
  `repl/completer.dart`); doc fix in `docs/spec/24_vault.md`
- **Dependencies:** none
- **Plan:**
  [plan_0_08_vault_file_export.md](../../plans/completed/plan_0_08_vault_file_export.md)
  (status: Complete). Implemented directly on `main`, no worktree/branch/PR
  (small, well-scoped plan). `vault export <uri> --output <path>` added;
  `--output` required (no stdout fallback, deliberate deviation from
  `vault get`); directory targets derive the filename from the manifest's
  `originalName`, sanitised via `p.basename(...).trim()` with a `blob`

## `kmdb_cli` cloud sync credentials: permission-hardened directory storage ✅ Complete

The Encryption confidentiality reconciliation work item (above) deliberately
scoped Google Drive OAuth credential storage out of its own remit (Q7):
`remote_config.dart` stored `AccessCredentials.toJson()` as plaintext JSON
under the per-machine `local/` directory, at the process umask (typically
`644`, world-readable) with no hardening at all. This was not a
database-encryption gap — the credentials never sync, are never written into
an SSTable, and have no relationship to the DEK/`enc:blob` even in principle —
but it was a genuine local-secret-at-rest exposure: anyone with filesystem
access to the CLI's `local/` directory could read a live OAuth token and act
as the user against their cloud storage, independent of whether the KMDB
database itself is encrypted. See §31 gap 9 (now resolved) for the
accepted-limitation framing this work item closed.

**Fix shipped:** the original plan for this work item chose OS-native
keychain integration (macOS Keychain, Windows Credential Manager, Linux
Secret Service) as the primary mechanism. During review, that pivoted to a
permission-hardened per-database directory instead — the same model OpenSSH
(`~/.ssh`, refuses to use a key file it finds group/world-readable) and
`gcloud` (`~/.config/gcloud` / `%APPDATA%\gcloud`) already use in production
for exactly this class of secret, and judged more proportionate than pulling
in three native backends for one CLI-managed credential. `local/` is `chmod
700`, each credential file `chmod 600`, both set at write time; the read path
hard-refuses (rather than silently proceeding) if either has drifted looser
than expected. Windows relies on default per-user-profile NTFS ACL
inheritance, no `icacls` shelling. Also fixed a leak found during
investigation: `remote remove` deleted the `config.json` entry but left the
credentials file itself behind.

OS-native keychain integration remains a possible future upgrade — the
package survey (`win32`/`dbus`/`security` CLI) that drove the original design
is preserved as a deferred item in [9_99.md](9_99.md) ("OS-native keychain
integration for `kmdb_cli` credentials").

- **Scope:** `kmdb_cli` (`credential_store.dart`, new
  `credential_store/directory_credential_store.dart`, `remote_command.dart`,
  `remote_config.dart`, `sync_command.dart`, `push_command.dart`,
  `pull_command.dart`); doc-comment update in `kmdb` core
  (`GoogleDriveRemoteConfig.credentialsPath`); new
  `docs/spec/33_cli_credential_store.md`
- **Dependencies:** none — independent of `kmdb` core and of the Encryption
  confidentiality reconciliation work item, which already shipped
- **Plan:**
  [plan_0_09_cli_keychain_credentials.md](../../plans/completed/plan_0_09_cli_keychain_credentials.md)
  (status: Complete). Implemented directly on `main`, no worktree/branch/PR
