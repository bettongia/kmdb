# v0.07 ✅ Complete

## Encryption ✅ Complete

Proposal: [Technical Proposal: Database Encryption](../proposals/encryption.md)

Plan: [Database Encryption (Phase 12)](../plans/completed/plan_encryption.md)
(PR [#46](https://github.com/bettongia/kmdb/pull/46))

Look at encryption options for the local and shared data.

Consider Per-platform secure storage (iOS Keychain, Android SharedPreferences,
web `localStorage`, desktop app-data dir) injectable via a `PlatformIdStore`
interface; default falls back to `$meta`.

Consider encryption at the value level in the KV store. This will need to be
very user-friendly and could follow an approach of the user selecting their own
salt rather than approaches such as managing PGP keys. The approach needs to
work in the environment where the user understands little about encryption and
won't store things like private keys right next to the KMDB data in their chosen
storage location (e.g. local directory or Google Drive)

Delivered in Phase 12 as opt-in, value-level AES-256-GCM encryption extending
the §5 value pipeline, with Argon2id KDF + envelope encryption (passphrase +
generated recovery code), vault blob encryption, and CLI support. See
[`docs/spec/31_encryption.md`](../spec/31_encryption.md) for the full design.

Follow-up deferred from Phase 12: the `kmdb_flutter` add-on package
(`FlutterSecureDekCache` for persistent DEK caching across sessions +
`cryptography_flutter` for native AES-256-GCM / Argon2id acceleration).
**Delivered** — see [`docs/plans/completed/plan_kmdb_flutter.md`](../plans/completed/plan_kmdb_flutter.md).
`FlutterSecureDekCache` is the **first consumer** of the `DekCache` interface
in a production-grade persistent storage scenario; the planned `PlatformIdStore`
abstraction (roadmap 0.07) will subsume it and provide a stable cross-path
device identifier. When `PlatformIdStore` lands, `FlutterSecureDekCache` becomes
its primary refactor target — see `plan_kmdb_flutter.md` F2 for the path-
stability caveat that `PlatformIdStore` is designed to resolve.

Side-effect of the encryption investigation (Q4): the §12 "Namespace-Scoped
Sync" documentation was corrected. There is no upload-time namespace filter —
system namespaces (`$index:`, `$fts:`, `$vec:`, `$ver:`, `$vault`) are written
to SSTables and reach the cloud via whole-file upload; their confidentiality
relies on value-level encryption when it is enabled.
