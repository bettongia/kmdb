# v? the unknown frontier

Unplanned things, strange things, hidden things - all need a home.

## Simple API

The API for the `kmdb` package is very comprehensive but it likely provides a
lot of interfaces that an application developer doesn't need. This plan is to
consider if an additional API can be provided that is a subset (or porcelain) of
the main API that will help an app developer get started.

## JSONPath — Deferred Extensions

KMDB currently supports a subset of RFC 9535 (JSONPath) for field path selectors
(see spec §13). The following extensions were explicitly deferred from the
initial implementation:

### Filter expressions

`$.policies[?(@.expired == true)]` — in-path filter predicates that select array
elements matching a condition. These overlap heavily with the existing Filter
DSL and require a unified expression layer design to avoid duplicating logic and
risking inconsistency. The right approach is to design a shared expression
evaluator first, then surface it in both the path syntax and the Filter DSL.

### Recursive descent

`$..name` — matches `name` at any depth in the document tree. Useful for deeply
nested or schema-less documents, but adds O(depth × fields) traversal cost with
non-obvious semantics for index fan-out (how many entries per document?
depth-bounded?). Deferred until there is a concrete use-case with known
performance requirements.

## Cross-collection / cross-document references

A future "foreign key" or join mechanism (e.g.
`orders[*].customerId -> customers`) would need path syntax to describe the join
key on both sides. The path grammar defined in §13 is designed to be composable
with such a feature: a reference expression could be expressed as two paths plus
a join operator, with each individual path using the existing subset. No grammar
changes are needed now, but this should be revisited when cross-collection query
design is planned.

## Cloud Adapter: Google Cloud Storage (low priority)

GCS will be the first object store adapter.

## Magika second-pass for vault text detection (opt-in)

**Deferred from the v0.06 roadmap (WI-7) on 2026-07-09** — pending an observed
misclassification issue from `betto_mediatype_detector` at vault ingest time.
No such issue has been seen to date, and the feature was already scoped as
opt-in/default-off, so there is no cost to deferring it until there's a
concrete case to justify the extra inference session. See the WI-7 entry in
[0.06 Roadmap](completed/0_06.md) for the original work-item history.

Add an optional Magika ONNX inference step inside the vault indexing isolate
(introduced in v0.06 WI-3) to validate whether a blob flagged as `text/plain`
is actually readable text. Surfaces the `is_text` flag and a confidence score;
would store `detectorScore` in the `$$vault:extract:{sha256}` KV entry (there
is no filesystem `extract_status.json` file — see the WI-10 investigation and
§32). `betto_mediatype_detector` remains the canonical detector at ingest
time — Magika would not modify `manifest.json`.

**Magika would be opt-in via `VaultSearchConfig` (default off).** This keeps
the default resource footprint unchanged, which matters on mobile. The
isolate would load and dispose the Magika session before loading the
embedding model — the two ORT sessions would never be resident
simultaneously.

A working Magika ONNX example already exists in `betto_onnxrt`; the model
itself is a few MB (significantly lighter than the embedding model).

- **Scope:** vault indexing isolate (native-only), `VaultSearchConfig` API
- **Dependencies:** v0.06 WI-3 (Complete)
- **Available:** Magika example in `betto_onnxrt` (needs surfacing as a
  callable API, not just a CLI demo)
- **Plan:** —

## `bge-m3` multilingual embedding model (deferred indefinitely)

**Deferred from the v0.06 roadmap (WI-4) on 2026-07-09** — `bge-small-en-v1.5`
(English) and `multilingual-e5-small` (~100 languages, WI-4) between them
cover the project's current language needs, so there's no near-term case to
absorb `bge-m3`'s extra complexity. Its own `ModelCatalog` stub
(`bge-m3-v1.0`, placeholder checksums, never downloadable) was removed
entirely during WI-4 rather than left in place — a greenfields project has no
cost to re-adding it properly whenever this work is actually picked up.

`bge-m3` is a bigger jump than "1024 dims, needs a re-index" alone suggests:
its fp32 ONNX export is ~2.29 GB total (`model.onnx`, 725 KB graph-only, plus
a separate `model.onnx_data`, 2.27 GB external weights) — it exceeds ONNX's
2 GB single-protobuf-file limit, so HuggingFace's export splits it into a
graph file plus an "external data" companion, a structurally different
layout `ModelSpec`/`ModelDownloader` don't support today (`ModelSpec.files`
assumes each named asset is one self-contained file with one checksum). Any
future pickup needs `ModelSpec`/`ModelDownloader` changes to fetch and place
a multi-file asset correctly, not just a new catalog entry with real
checksums.

- **Scope:** `betto_inferencing` (`ModelSpec`/`ModelDownloader` multi-file
  asset support, then a real `ModelCatalog` registration)
- **Dependencies:** none blocking; needs the multi-file asset support built
  first
- **Available:** full trade-off analysis in
  [plan_0_06_wi4_multilingual_embedding_model.md](../plans/completed/plan_0_06_wi4_multilingual_embedding_model.md)'s
  Q5
- **Plan:** —

## Quantized/GPU-optimized ONNX variants for embedding models (deferred)

**Deferred from the v0.06 roadmap (WI-4) on 2026-07-09** — trading a model's
plain fp32 export for a hardware-specific quantized/optimized one changes
retrieval accuracy and needs its own dedicated validation pass; not worth
picking up until first-use download size on mobile proves a real problem.

`multilingual-e5-small` (and likely other future HuggingFace-sourced models)
publish size-reducing alternatives to the plain fp32 `model.onnx` this
project registers — e.g. `model_qint8_avx512_vnni.onnx` (x86 AVX512-VNNI
int8, a poor fit for this project's largely-ARM targets) and `model_O4.onnx`
(fp16 mixed precision intended for GPU inference, not the CPU-only execution
providers `betto_onnxrt` supports). Neither is a good fit for this project's
current runtime targets, but a smaller download could matter on mobile if
first-use size becomes a real complaint.

- **Scope:** `betto_inferencing` (`ModelCatalog`, possibly a second, opt-in
  catalog entry per model)
- **Dependencies:** none
- **Available:** —
- **Plan:** —
