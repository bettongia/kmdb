## Threat-model decision (Group C) ✅ Decided

The review's **E-1** established that `docs/spec/31_encryption.md` documents a
**passive-only** threat model — a provider who *reads* your data — and never
claims the synced data is authentic. Five of the six critical findings follow
from that single gap.

**Decision (2026-07-19): close the gap before `0.1.0`, scoped to T1.**

- **T1 (untrusted provider, read *and* write) — closed in this track.** All sync
  artefacts are authenticated. The provider cannot forge SSTables, vault blobs,
  leases, or high-water marks. The realistic threat this defeats is **account
  compromise** — a phished Google account or stolen OAuth token yields write
  access to the sync folder, and neither full-disk encryption nor the provider's
  own at-rest encryption helps there.
- **T3 (malicious peer) — designed, deferred.** A compromised peer already holds
  the DEK and full plaintext, so authentication buys attribution and blast-radius
  containment, not confidentiality. It needs per-device asymmetric identity,
  enrollment, and revocation. Captured as a proposal, not built here.

**Authentication is decoupled from encryption.** Encryption stays optional and is
sold on its own merits. The reasoning: with full-disk encryption and a reputable
provider, KMDB-level encryption is genuinely marginal for many users — so
forcing it as a *prerequisite* for authentication would trade a low-value
requirement for a high-value guarantee. The sync-auth key is also a
fundamentally lower-stakes secret than the DEK: losing it means re-provisioning,
not losing your data, so it needs no passphrase and no recovery code.

### WI-11: SC-10 — index state syncs, and a second device silently returns nothing 🔴

**Priority 1. Code fix, and it changes a `$meta` key layout — so it must land
before the `0.1.0` format freeze.**

Found by the second W1 pass and **reproduced end to end**:

```
PROBE A final: ScanStrategy.indexScan, 3 rows
PROBE B:       ScanStrategy.indexScan, 0 rows   (expected 3)
PROBE B unfiltered scan: 3 docs
```

The chain, independently verified:

1. §16:229–230 claims *"the sync engine filters out all `$`-prefixed namespaces
   during SSTable upload, so index state never leaves the device."*
   **No such filter has ever existed** — `git log -S` finds it at no commit. The
   only filter is file-level `localOnly`, driven by `isLocalOnly`, which matches
   `$$` **only**. §12:455 correctly documents `$meta` as uploaded; the two specs
   contradict and §12 is right.
2. Index state — including `status` — lives in `$meta`
   ([index_manager.dart:70](../../packages/kmdb/lib/src/query/index/index_manager.dart#L70)).
3. Device B pulls A's `$meta` and inherits `status: current` for an index **it
   never built**. `_nameToKey` is a plain XXH64, so it is device-independent, and
   `builtThrough`/`gen:{ns}` sync together and validate each other.
4. B runs an index scan against its own empty `$$index:*` namespaces and returns
   **zero rows for documents that are present and match** — silently, with
   `QueryPlan` reporting `indexScan`.

It self-heals once B *writes* to that namespace, **so a read-mostly second device
is worst affected** — a phone reading what a laptop writes is exactly the shape.

> **Severity note.** The taxonomy reserves 🔴 for data loss, corruption, or a
> security break, and strictly nothing is lost here. Promoted anyway: a silently
> wrong result on the primary read path, with no error and no workaround, is at
> least as damaging as detectable corruption — and unlike corruption the user has
> no signal that anything is wrong. 🟠 is defensible; it blocks either way.

#### ⚠️ Do not "add the missing filter" — it is incoherent, not merely absent

The namespace rule is:

| Prefix | Meaning | Syncs? |
| :--- | :--- | :---: |
| `$` | System namespace | **Yes** |
| `$$` | System **and local-only** | No — `.local.sst`, never uploaded |

`namespace_codec.dart` states that **"the `$$` prefix is a strict superset of the
`$` prefix"**, and the single-`$` namespaces actually in use are `$meta`, `$ver`,
`$vault` and `$sync`.

**So implementing §16's claimed filter would break sync outright** — it would
exclude `$meta` (device identity, generation counters, the wrapped DEK), `$ver`
(version history) and `$vault` (blob references) from upload. The mechanism §16
describes could never have worked; it is not a filter someone forgot to write.

**The actual defect is that index status is device-local state stored in a synced
namespace.** The fix is to move it — to a `$$` namespace, or to a per-device key
within `$meta` — **not** to filter `$`.

#### SC-15 — the same defect, one layer down (added 2026-07-20)

**A case-insensitive equality filter returns zero rows once an index exists on
the field.** Reproduced; the only variable is whether an index was declared:

```
no index declared  -> rows = 1
index declared     -> rows = 0   strategy=indexScan scanned=0 matched=0
control exact case -> rows = 1   strategy=indexScan
```

`FieldFilter.equalityPredicate` is `_op == _Op.eq ? (_path, _operand) : null`
([field_filter.dart:175](../../packages/kmdb/lib/src/query/filter/field_filter.dart#L175))
— it never consults `caseSensitive`, which is a field on the *same class* four
lines above. So the planner is handed an exact-match predicate, does an index
lookup on the exact token, and the in-memory pass §13:507 promises as the safety
net never runs, because the index "already answered."

**SC-10 and SC-15 are one root cause twice: the planner trusting a predicate it
has not established the index can answer.** SC-10 crosses a device boundary,
SC-15 a case-folding boundary. Fix them together — and note that the one-line
fix is narrow: **any future non-exact matcher (locale folding, accent
insensitivity, normalisation) reintroduces this verbatim** unless the planner is
made to ask whether the index can answer *this* predicate, rather than whether
the predicate is an equality.

**Why no test caught it:** `filter_test.dart:173–196` covers `caseSensitive`
thoroughly at the `evaluate()` level, and `index_query_test.dart` covers index
selection thoroughly with exact-case predicates. Each half is well tested and
both score well on line coverage. **The seam between them is untested — no test
anywhere asserts that the index path and the full-scan path return the same
rows.** That equivalence property is the test this WI must add.

#### The broader audit this WI must include

Nobody has audited **which `$meta` entries are device-local facts and which are
replicated state**. Every entry needs classifying, and the answer is not obvious
for several:

| `$meta` entry | Device-local or replicated? |
| :--- | :--- |
| Index state (`status`, `builtThrough`) | **Device-local** — this is SC-10 |
| Device identity | **Device-local**, yet synced — SC-5, same bug class. Also the collision that sank the sync-set-identity proposal |
| `enc:blob` (wrapped DEK) | **Replicated** — correct as-is |
| `gen:{ns}` generation counters | **Unclear** — needs a decision, since cache invalidation depends on it |
| Tombstone GC floor (H4-FU3) | **Unclear** — one device's floor affecting another's ingest may or may not be intended |

Two of the five are known-wrong; two are undecided. That is the real scope of
this work item.

**How it survived is the transferable lesson.** The claim was false the day it
was written. WI-0 later built the real `$$`/`.local.sst` mechanism and rewrote
the *adjacent bullet in the same paragraph*, leaving this one standing on the
rationale it had just invalidated. The section reads as freshly maintained
because in part it is — which defeats every heuristic in the review's §11.1:
not stale prose, not a duplicated definition, not present-tense design intent.

> ### ✅ SC-3 decided (2026-07-20): correct §15; do not build it
>
> **Correct all five sites** to state plainly that the materialised view cache is
> **not implemented**, and **drop the "Required" marking** from §15's mobile and
> web tier rows.
>
> Three things drove this:
>
> 1. **The specified design is wrong, not merely unbuilt.** §15 specifies
>    `$cache:{namespace}:{queryHash}` — **single `$`, so it syncs.** A
>    materialised view cache is derived, device-local state by definition;
>    putting it in a synced namespace is exactly the mistake **SC-10** embodies.
>    Building it as written would give device B device A's cached query results.
> 2. **The problem may not exist.** §18 measures `Scan (namespace, 100 results)`
>    at **7.43ms P99**. §15's rationale is that mobile and web processes are
>    killed silently and "must rebuild from `$cache` on cold open" — but the
>    rebuild appears to cost tens of milliseconds, not seconds.
> 3. **Nobody has missed it.** `CacheLayer.scan` has deferred to "the Query Layer
>    (Phase 7, `KmdbQuery`)" since Phase 7 — through the vault, encryption, and
>    search work — and `KmdbQuery` never picked it up.
>
> **Make the correction do work.** Rather than deleting the claim, record the
> evidence that would justify building it later: *measure cold-start scan cost on
> a real mobile device and on web; build only if it exceeds a stated threshold.*
> That keeps the decision revisitable on evidence instead of losing the intent.
>
> **If it is ever built, it goes in `$$cache`, not `$cache`** — decided at the
> same time. Derived, device-local state belongs behind the `$$` prefix so it
> lands in `.local.sst` and never uploads, consistent with `$$fts:`, `$$vec:`
> and `$$index:`. §15's current single-`$` naming must not survive the
> correction, or the next reader takes it at face value.

  the project's own stance in `plan_vault_gc_failsafe.md`, and is unverified on
  web where `int` semantics differ.
- **S-8** — size, depth, and time bounds on the PDF/HTML/Markdown extractors.
- **F-1** — move native allocations inside the `try` in `betto_zstd.compress`
  and `betto_icu`.

### WI-7: QA advisories from the PR #61 review (2026-07-20)

Advisory items deliberately **not** actioned in PR #61, to keep that change
scoped:

- **A3 — quarantine is invisible to the embedding application.** `pull()`
  returns `Future<void>` and reports a skipped peer file only via `print`. This
  matters more than its severity suggests: quarantine is per-spec, but because
  `consolidate()` now skips the file too, a *legitimately* corrupt file — no
  attacker involved — is **permanently skipped by this device**, and the host
  application has no programmatic way to learn its peer's data was dropped.
  **Deserves its own plan**; surface it in a result object rather than a
  console print.
- **A5 — breaking public API changes with no CHANGELOG note.**
  `ConsolidationCoordinator` is exported from `kmdb.dart` and gained a *required*
  `dbDir`; vault blobs gained an unconditional 1-byte envelope prefix. Both are
  fine pre-publication under the no-migration decision, but the CHANGELOG still
  reads only "Initial pub.dev release." Fold into Group D before tagging.
- **A6 — confirm the Windows staging path under RC-3.** `_stagingDir` builds
  `'$dbDir/tmp/consolidation'` with a literal `/` in a fix whose stated purpose
  was Windows compatibility. It matches the existing `VaultStore.stagingDir`
  precedent and Dart's `File` accepts forward slashes on Windows, so this is
  consistency rather than drift — but it wants confirming on a real Windows host.
- **A2 — `close()` can throw after a successful flush.**
  `_vaultSearchManager?.close()` is the last statement and unguarded. The
  reordering delivered the durability guarantee; wrapping this call would deliver
  the remainder cheaply.

### WI-5: SC-1 unlock policy

**Blocked on decisions in the proposal's §9** — agent DEK lifetime, WebAuthn PRF
browser support, the default re-authentication interval, and whether
per-operation escalation is in scope.

[Proposal](../proposals/unlock_policy.md). SC-1 is that **a wrong passphrase
opens an encrypted database whenever the DEK cache is warm** — the recommended
mobile configuration. The proposal replaces the raw-DEK cache with 1Password's
wrapped-copy model, which makes the bug class structurally impossible rather
than patching it, and adds a CLI session agent and a headless-server path.

> **§31's own stated remedy does not fix SC-1.** Confirming the cached DEK
> against `enc:blob` proves the *DEK* is valid; the passphrase never enters the
> cache-hit path. **This must not be resolved by editing §31 to match the code.**

### WI-8: `betto_zstd` frame-size cap — closes S-2

External repo, own PR and publish. See the S-2 caveat above for why the
KMDB-side cap is insufficient on its own.

**Must land before WI-9**, since `betto_zstd` is one of the twelve packages the
release gate publishes — otherwise its `0.1.0` is burned and needs republishing.

Also fold in **F-1** while there (native allocations before the `try` in
`compress`), since it is the same repo and the same PR.
