# v0.10.01

## Purpose

This is a **hardening track**, following the precedent set by
[0_02_01](completed/0_02_01.md) — the roadmap that closed out the durability
findings of the 2026-05-22 code review. It exists to work through the findings
of the **2026-07-18 release-readiness review**
([docs/reviews/release-readiness-review-2026-07-18.md](../reviews/release-readiness-review-2026-07-18.md)),
which recommended **against tagging `0.1.0`** in the codebase's current state.

> ### ⚠️ Callback to the review — do not close this track early
>
> **The review is the authoritative status board for this work.** Its §9
> "Coverage actually achieved" records exactly which workstreams ran and to what
> depth. As of 2026-07-20:
>
> | Workstream | State |
> | :--- | :--- |
> | W2 security, W3 FFI | ✅ Complete |
> | **W4** concurrency/durability | ✅ **Complete** (2026-07-20) — found **D-1**; confirmed the 0.02.01 hardening held |
> | **W1** spec conformance | ⚠️ **~40%** — tiers 1–3 audited, **tiers 4–6 never traced** (see WI-1) |
> | W5 CLI, W6 code health | ⚠️ **Partial** — see §9 for what was never exercised |
>
> **Every item in the review must be worked through before `0.1.0` is tagged,
> not merely the criticals closed by PR #61.** The review's §6 findings register
> is the checklist; its §10 groups findings by shared remediation work.
>
> **The specific trap this callback exists to prevent:** W1 found a spec section
> marking a feature **Required on mobile and web** that does not exist at all
> (SC-3), and two specs giving **contradictory byte layouts** for a format
> `0.1.0` freezes (SC-2). Both were in the 40% that got looked at. The untraced
> 60% has had no such scrutiny.

> ### ⚠️ S-2 is only partially closed — do not tick it (this is **WI-8**)
>
> Raised by `kmdb-qa` on 2026-07-20 during the PR #61 review, and recorded here
> because WI-0 would otherwise read as closing S-2 outright.
>
> The KMDB-side cap (`ValueCodec.kMaxDecodedValueBytes`, 1 MiB) fires **after**
> `decompress()` returns. That bounds a survivable-but-oversized value, and the
> per-document guards in `dump`/`export`/`scan` mean one poisoned value can no
> longer abort a whole collection — both real improvements. But it does **not**
> mitigate S-2's headline attack: a Zstd frame *declaring* a huge decompressed
> size still reaches `malloc` unbounded, because that check belongs upstream in
> `betto_zstd` (a separate repo, its own PR and publish, gated behind Group D).
>
> `docs/spec/05_value_encoding.md` states this limitation accurately. **Mark S-2
> closed only when both halves have landed.**

## Threat-model decision (Group C) ✅ Decided

The review's **E-1** established that `docs/spec/31_encryption.md` documents a
**passive-only** threat model — a provider who *reads* your data — and never
claims the synced data is authentic. Five of the six critical findings follow
from that single gap.

**Decision (2026-07-19): close the gap before `0.1.0`, scoped to T1.**

- **T1 (untrusted provider, read *and* write) — closed in this track.** All sync
  artefacts are authenticated. The provider cannot forge SSTables, vault blobs,
  leases, or high-water marks. The realistic threat this defeats is **account
  compromise** — a phished Google account or stolen OAuth token yields write
  access to the sync folder, and neither full-disk encryption nor the provider's
  own at-rest encryption helps there.
- **T3 (malicious peer) — designed, deferred.** A compromised peer already holds
  the DEK and full plaintext, so authentication buys attribution and blast-radius
  containment, not confidentiality. It needs per-device asymmetric identity,
  enrollment, and revocation. Captured as a proposal, not built here.

**Authentication is decoupled from encryption.** Encryption stays optional and is
sold on its own merits. The reasoning: with full-disk encryption and a reputable
provider, KMDB-level encryption is genuinely marginal for many users — so
forcing it as a *prerequisite* for authentication would trade a low-value
requirement for a high-value guarantee. The sync-auth key is also a
fundamentally lower-stakes secret than the DEK: losing it means re-provisioning,
not losing your data, so it needs no passphrase and no recovery code.

### WI-1: Complete W1 — symbol-grep sweep of spec tiers 4–6

**Priority 1. No prerequisites. Do this before scoping anything else.**

W1 audited tiers 1–3 (encryption, vault, sync, the on-disk formats, the
durability contract). **Tiers 4–6 were never traced:** §10, §11, §13, §16, §22,
§25, §26, §29, §30, §32, §33. §12 and §24 got targeted passes on
encryption/integrity claims only.

The `kmdb-architect` pass identified that the findings share three mechanical
causes, which makes the remainder cheap to sweep:

1. **Duplicated byte layouts** — the same format documented in two sections that
   then drift (this is SC-2).
2. **Spec-named API symbols with nothing binding them to reality** — §15 names
   `$cache` and `KvStoreConfig.sessionCacheMaxObjects`; **neither exists** (SC-3).
3. **Design intent written in the present tense** — reads as implemented.

**Method:** grep every spec-named symbol, namespace, config field and exception
type against the codebase. That single mechanical pass would have caught SC-3
directly, and is far cheaper than re-reading 6,000 lines of prose.

- [ ] Extract every code identifier named in tiers 4–6 and grep it
- [ ] Trace surviving normative claims to code + test
- [ ] Record findings in the review's §6 register and §11 matrix
- [ ] Update review §9 to mark W1 complete

### WI-2: Spec corrections — SC-2 and SC-3

**Priority 2. Soft prerequisite: WI-1** (the sweep may lengthen the list).

**SC-2 🟠 — two specs give contradictory byte layouts, and the security document
is the stale one.** §31 still documents the vault blob format PR #61 replaced,
*and* still describes the S-4-vulnerable `manifest.encrypted` read gate as
current behaviour. §24 is correct. An integrator reading §31 implements the
wrong format **and** the wrong trust model, for a format `0.1.0` freezes.

- [ ] Make §24 the single authority for the vault blob layout
- [ ] Delete the duplicated layout prose from §31; cross-reference instead
- [ ] Remove §31's stale `manifest.encrypted` gate description

**SC-3 🟠 — needs a decision, not a fix.** §15's Materialised View Cache does not
exist: no read or write of `$cache` anywhere in `lib`, and
`KvStoreConfig.sessionCacheMaxObjects` is absent too. `CacheLayer.scan` defers it
to `KmdbQuery`, which never picked it up — a hand-off that reads as "implemented
elsewhere" from both ends. **§15's tier table marks it Required on mobile and
web.**

> **Decision required before work starts:** build the materialised view cache, or
> correct §15 to state it is not implemented and drop the Required marking.
> Building it is real work with a real justification (mobile and web processes
> are killed silently, which is why §15 wanted it). Correcting the spec is
> cheap but removes a documented capability.

### WI-7: QA advisories from the PR #61 review (2026-07-20)

Advisory items deliberately **not** actioned in PR #61, to keep that change
scoped:

- **A3 — quarantine is invisible to the embedding application.** `pull()`
  returns `Future<void>` and reports a skipped peer file only via `print`. This
  matters more than its severity suggests: quarantine is per-spec, but because
  `consolidate()` now skips the file too, a *legitimately* corrupt file — no
  attacker involved — is **permanently skipped by this device**, and the host
  application has no programmatic way to learn its peer's data was dropped.
  **Deserves its own plan**; surface it in a result object rather than a
  console print.
- **A5 — breaking public API changes with no CHANGELOG note.**
  `ConsolidationCoordinator` is exported from `kmdb.dart` and gained a *required*
  `dbDir`; vault blobs gained an unconditional 1-byte envelope prefix. Both are
  fine pre-publication under the no-migration decision, but the CHANGELOG still
  reads only "Initial pub.dev release." Fold into Group D before tagging.
- **A6 — confirm the Windows staging path under RC-3.** `_stagingDir` builds
  `'$dbDir/tmp/consolidation'` with a literal `/` in a fix whose stated purpose
  was Windows compatibility. It matches the existing `VaultStore.stagingDir`
  precedent and Dart's `File` accepts forward slashes on Windows, so this is
  consistency rather than drift — but it wants confirming on a real Windows host.
- **A2 — `close()` can throw after a successful flush.**
  `_vaultSearchManager?.close()` is the last statement and unguarded. The
  reordering delivered the durability guarantee; wrapping this call would deliver
  the remainder cheaply.

### WI-5: SC-1 unlock policy

**Blocked on decisions in the proposal's §9** — agent DEK lifetime, WebAuthn PRF
browser support, the default re-authentication interval, and whether
per-operation escalation is in scope.

[Proposal](../proposals/unlock_policy.md). SC-1 is that **a wrong passphrase
opens an encrypted database whenever the DEK cache is warm** — the recommended
mobile configuration. The proposal replaces the raw-DEK cache with 1Password's
wrapped-copy model, which makes the bug class structurally impossible rather
than patching it, and adds a CLI session agent and a headless-server path.

> **§31's own stated remedy does not fix SC-1.** Confirming the cached DEK
> against `enc:blob` proves the *DEK* is valid; the passphrase never enters the
> cache-hit path. **This must not be resolved by editing §31 to match the code.**

### WI-8: `betto_zstd` frame-size cap — closes S-2

External repo, own PR and publish. See the S-2 caveat above for why the
KMDB-side cap is insufficient on its own.

**Must land before WI-9**, since `betto_zstd` is one of the twelve packages the
release gate publishes — otherwise its `0.1.0` is burned and needs republishing.

Also fold in **F-1** while there (native allocations before the `try` in
`compress`), since it is the same repo and the same PR.
