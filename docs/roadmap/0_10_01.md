## Threat-model decision (Group C) ‚úÖ Decided

The review's **E-1** established that `docs/spec/31_encryption.md` documents a
**passive-only** threat model ‚Äî a provider who *reads* your data ‚Äî and never
claims the synced data is authentic. Five of the six critical findings follow
from that single gap.

**Decision (2026-07-19): close the gap before `0.1.0`, scoped to T1.**

- **T1 (untrusted provider, read *and* write) ‚Äî closed in this track.** All sync
  artefacts are authenticated. The provider cannot forge SSTables, vault blobs,
  leases, or high-water marks. The realistic threat this defeats is **account
  compromise** ‚Äî a phished Google account or stolen OAuth token yields write
  access to the sync folder, and neither full-disk encryption nor the provider's
  own at-rest encryption helps there.
- **T3 (malicious peer) ‚Äî designed, deferred.** A compromised peer already holds
  the DEK and full plaintext, so authentication buys attribution and blast-radius
  containment, not confidentiality. It needs per-device asymmetric identity,
  enrollment, and revocation. Captured as a proposal, not built here.

**Authentication is decoupled from encryption.** Encryption stays optional and is
sold on its own merits. The reasoning: with full-disk encryption and a reputable
provider, KMDB-level encryption is genuinely marginal for many users ‚Äî so
forcing it as a *prerequisite* for authentication would trade a low-value
requirement for a high-value guarantee. The sync-auth key is also a
fundamentally lower-stakes secret than the DEK: losing it means re-provisioning,
not losing your data, so it needs no passphrase and no recovery code.

### WI-11: SC-10 ‚Äî index state syncs, and a second device silently returns nothing üî¥

**Priority 1. Code fix, and it changes a `$meta` key layout ‚Äî so it must land
before the `0.1.0` format freeze.**

Found by the second W1 pass and **reproduced end to end**:

```
PROBE A final: ScanStrategy.indexScan, 3 rows
PROBE B:       ScanStrategy.indexScan, 0 rows   (expected 3)
PROBE B unfiltered scan: 3 docs
```

The chain, independently verified:

1. ¬ß16:229‚Äì230 claims *"the sync engine filters out all `$`-prefixed namespaces
   during SSTable upload, so index state never leaves the device."*
   **No such filter has ever existed** ‚Äî `git log -S` finds it at no commit. The
   only filter is file-level `localOnly`, driven by `isLocalOnly`, which matches
   `$$` **only**. ¬ß12:455 correctly documents `$meta` as uploaded; the two specs
   contradict and ¬ß12 is right.
2. Index state ‚Äî including `status` ‚Äî lives in `$meta`
   ([index_manager.dart:70](../../packages/kmdb/lib/src/query/index/index_manager.dart#L70)).
3. Device B pulls A's `$meta` and inherits `status: current` for an index **it
   never built**. `_nameToKey` is a plain XXH64, so it is device-independent, and
   `builtThrough`/`gen:{ns}` sync together and validate each other.
4. B runs an index scan against its own empty `$$index:*` namespaces and returns
   **zero rows for documents that are present and match** ‚Äî silently, with
   `QueryPlan` reporting `indexScan`.

It self-heals once B *writes* to that namespace, **so a read-mostly second device
is worst affected** ‚Äî a phone reading what a laptop writes is exactly the shape.

> **Severity note.** The taxonomy reserves üî¥ for data loss, corruption, or a
> security break, and strictly nothing is lost here. Promoted anyway: a silently
> wrong result on the primary read path, with no error and no workaround, is at
> least as damaging as detectable corruption ‚Äî and unlike corruption the user has
> no signal that anything is wrong. üüÝ is defensible; it blocks either way.

#### ‚öÝÔ∏è Do not "add the missing filter" ‚Äî it is incoherent, not merely absent

The namespace rule is:

| Prefix | Meaning | Syncs? |
| :--- | :--- | :---: |
| `$` | System namespace | **Yes** |
| `$$` | System **and local-only** | No ‚Äî `.local.sst`, never uploaded |

`namespace_codec.dart` states that **"the `$$` prefix is a strict superset of the
`$` prefix"**, and the single-`$` namespaces actually in use are `$meta`, `$ver`,
`$vault` and `$sync`.

**So implementing ¬ß16's claimed filter would break sync outright** ‚Äî it would
exclude `$meta` (device identity, generation counters, the wrapped DEK), `$ver`
(version history) and `$vault` (blob references) from upload. The mechanism ¬ß16
describes could never have worked; it is not a filter someone forgot to write.

**The actual defect is that index status is device-local state stored in a synced
namespace.** The fix is to move it ‚Äî to a `$$` namespace, or to a per-device key
within `$meta` ‚Äî **not** to filter `$`.

#### SC-15 ‚Äî the same defect, one layer down (added 2026-07-20)

**A case-insensitive equality filter returns zero rows once an index exists on
the field.** Reproduced; the only variable is whether an index was declared:

```
no index declared  -> rows = 1
index declared     -> rows = 0   strategy=indexScan scanned=0 matched=0
control exact case -> rows = 1   strategy=indexScan
```

`FieldFilter.equalityPredicate` is `_op == _Op.eq ? (_path, _operand) : null`
([field_filter.dart:175](../../packages/kmdb/lib/src/query/filter/field_filter.dart#L175))
‚Äî it never consults `caseSensitive`, which is a field on the *same class* four
lines above. So the planner is handed an exact-match predicate, does an index
lookup on the exact token, and the in-memory pass ¬ß13:507 promises as the safety
net never runs, because the index "already answered."

**SC-10 and SC-15 share a shape, not a fix** (plan review, 2026-07-21): both are
the planner trusting a predicate it has not established the index can answer ‚Äî
SC-10 across a device boundary, SC-15 across a case-folding boundary ‚Äî but their
fixes touch disjoint files (a namespace move vs a one-line predicate gate). They
are bundled because they share one test artefact: the equivalence property that
index-path rows must equal full-scan rows. **The SC-15 fix is deliberately
narrow** (gate `equalityPredicate` on `caseSensitive`, plus a doc-comment
contract); the durable guard against a future non-exact matcher (locale folding,
accent insensitivity, normalisation) reintroducing it is that equivalence test,
not a capability contract ‚Äî see the plan's Q-A decision.

**Why no test caught it:** `filter_test.dart:173‚Äì196` covers `caseSensitive`
thoroughly at the `evaluate()` level, and `index_query_test.dart` covers index
selection thoroughly with exact-case predicates. Each half is well tested and
both score well on line coverage. **The seam between them is untested ‚Äî no test
anywhere asserts that the index path and the full-scan path return the same
rows.** That equivalence property is the test this WI must add.

### The `$meta` end-state (shared reference for WI-11 + WI-12 + WI-13)

Three WIs touch `$meta`; this is the target they converge on. The governing
invariant: **`$meta` holds only replicated state ‚Äî every device-local fact moves
to a `$$` namespace that lands in `.local.sst` and never uploads.** That turns the
`$` vs `$$` prefix rule from a mechanical accident (true by luck, violated by five
entries ‚Äî SC-10 is what that costs) into a *semantic* rule the next `$meta`
addition has to pass.

| Entry (symbolic key) | Written by | Classification | To-be location | Driver |
| :--- | :--- | :--- | :--- | :--- |
| `index:{ns}:{path}` | secondary-index state | device-local | **‚Üí `$$indexstate`** | WI-11 |
| `fts:{ns}:{field}` | FTS index state | device-local | **‚Üí `$$ftsstate`** | WI-11 |
| `vec:{ns}:{field}` | Vec index state | device-local | **‚Üí `$$vecstate`** | WI-11 |
| `gc:tombstoneFloor` | tombstone GC floor | device-local | **‚Üí `$$` (name unpinned)** | WI-11 |
| `device_id` | device identity | device-local (authoritative store is the local `DEVICE_ID` file; `$meta` copy is legacy/inert) | **retire the inert `$meta` copy** (see note) | WI-12 |
| `gen:{ns}` | generation counters | **undecided** | device-local `$$` **or** stays (max-merge) | WI-13 |
| `dirty` | dirty-open flag | device-local (expected) | **verify** ‚Äî stays or moves | WI-11 Phase 3 |
| `enc:blob` | wrapped DEK | replicated | **stays** | ‚Äî |
| `schema:{collection}` + `schema:__registry__` | schema state | replicated | **stays** | ‚Äî |
| `version:config:{collection}` | version retention policy | replicated | **stays** | ‚Äî |
| `namespaces` | namespaces registry | replicated | **stays** | ‚Äî |
| `formatVersion` | format-version marker | replicated | **stays** | ‚Äî |

**Resulting `$meta`** (after all three land): `enc:blob`, `schema:*`,
`version:config:*`, `namespaces`, `formatVersion` ‚Äî the state that *should* be
identical on every device that syncs the set. Everything else is device-local.

Caveats worth holding:

- **`gen:{ns}` is the swing entry.** The clean "purely replicated" invariant holds
  only if WI-13 lands it device-local. If it stays replicated-with-max-merge, it
  remains in `$meta` but needs merge semantics `$meta`'s plain LWW lacks today.
- **The `dirty` flag is unverified.** Device-local by nature; if Phase 3 finds it
  currently syncs, that is a further SC-10-class instance and it moves too.
- **The tombstone floor's `$$` namespace has no pinned name yet** ‚Äî unlike the
  three index states, it has no data-namespace sibling to mirror (it is one
  per-device key). WI-11 must pick: its own `$$` namespace or a shared local-meta
  one.
- **`device_id` is already authoritatively device-local ‚Äî WI-12 is a cleanup, not
  a migration (corrected 2026-07-21, verified against code + the `demodb` instance).**
  `ensureDeviceId` ([kv_store_impl.dart:407](../../packages/kmdb/lib/src/engine/kvstore/kv_store_impl.dart#L407))
  reads a **plaintext `DEVICE_ID` file in the db root first** (outside `sst/`, never
  synced) and only falls back to `$meta` for backward compatibility; its own doc
  comment says the file "is always preferred over the `$meta` value when both are
  present." So the authoritative identity is *already* outside the synced database.
  The `$meta` copy is written on first open and **does** sync, but is **inert on
  read**. Consequences: (a) **SC-5's severity drops** ‚Äî the exposure is
  hygiene/confidentiality (the copy leaks each peer's `device_id` and is dead weight),
  not a wrong-identity correctness bug; (b) **WI-12 = stop writing the inert `$meta`
  copy** (keep the legacy read-fallback), a small isolated change that must **not**
  touch the load-bearing file‚Üínaming‚Üímanifest chain; (c) ¬ß08's "must not be stored
  inside the database" is **substantially already honoured** by the file ‚Äî the
  Keychain/secure-storage move is a *separate optional enhancement*, and
  `device_id.dart:37-38`'s "`$meta` is the sole persistence mechanism" is a stale
  claim to log under WI-2.

> ### ‚úÖ SC-3 decided (2026-07-20): correct ¬ß15; do not build it
>
> **Correct all five sites** to state plainly that the materialised view cache is
> **not implemented**, and **drop the "Required" marking** from ¬ß15's mobile and
> web tier rows.
>
> Three things drove this:
>
> 1. **The specified design is wrong, not merely unbuilt.** ¬ß15 specifies
>    `$cache:{namespace}:{queryHash}` ‚Äî **single `$`, so it syncs.** A
>    materialised view cache is derived, device-local state by definition;
>    putting it in a synced namespace is exactly the mistake **SC-10** embodies.
>    Building it as written would give device B device A's cached query results.
> 2. **The problem may not exist.** ¬ß18 measures `Scan (namespace, 100 results)`
>    at **7.43ms P99**. ¬ß15's rationale is that mobile and web processes are
>    killed silently and "must rebuild from `$cache` on cold open" ‚Äî but the
>    rebuild appears to cost tens of milliseconds, not seconds.
> 3. **Nobody has missed it.** `CacheLayer.scan` has deferred to "the Query Layer
>    (Phase 7, `KmdbQuery`)" since Phase 7 ‚Äî through the vault, encryption, and
>    search work ‚Äî and `KmdbQuery` never picked it up.
>
> **Make the correction do work.** Rather than deleting the claim, record the
> evidence that would justify building it later: *measure cold-start scan cost on
> a real mobile device and on web; build only if it exceeds a stated threshold.*
> That keeps the decision revisitable on evidence instead of losing the intent.
>
> **If it is ever built, it goes in `$$cache`, not `$cache`** ‚Äî decided at the
> same time. Derived, device-local state belongs behind the `$$` prefix so it
> lands in `.local.sst` and never uploads, consistent with `$$fts:`, `$$vec:`
> and `$$index:`. ¬ß15's current single-`$` naming must not survive the
> correction, or the next reader takes it at face value.

### WI-7: QA advisories from the PR #61 review (2026-07-20)

Advisory items deliberately **not** actioned in PR #61, to keep that change
scoped:

- **A3 ‚Äî quarantine is invisible to the embedding application.** `pull()`
  returns `Future<void>` and reports a skipped peer file only via `print`. This
  matters more than its severity suggests: quarantine is per-spec, but because
  `consolidate()` now skips the file too, a *legitimately* corrupt file ‚Äî no
  attacker involved ‚Äî is **permanently skipped by this device**, and the host
  application has no programmatic way to learn its peer's data was dropped.
  **Deserves its own plan**; surface it in a result object rather than a
  console print.
- **A5 ‚Äî breaking public API changes with no CHANGELOG note.**
  `ConsolidationCoordinator` is exported from `kmdb.dart` and gained a *required*
  `dbDir`; vault blobs gained an unconditional 1-byte envelope prefix. Both are
  fine pre-publication under the no-migration decision, but the CHANGELOG still
  reads only "Initial pub.dev release." Fold into Group D before tagging.
- **A6 ‚Äî confirm the Windows staging path under RC-3.** `_stagingDir` builds
  `'$dbDir/tmp/consolidation'` with a literal `/` in a fix whose stated purpose
  was Windows compatibility. It matches the existing `VaultStore.stagingDir`
  precedent and Dart's `File` accepts forward slashes on Windows, so this is
  consistency rather than drift ‚Äî but it wants confirming on a real Windows host.
- **A2 ‚Äî `close()` can throw after a successful flush.**
  `_vaultSearchManager?.close()` is the last statement and unguarded. The
  reordering delivered the durability guarantee; wrapping this call would deliver
  the remainder cheaply.

### WI-5: SC-1 unlock policy

**Blocked on decisions in the proposal's ¬ß9** ‚Äî agent DEK lifetime, WebAuthn PRF
