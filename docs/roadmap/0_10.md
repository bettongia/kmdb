# v0.10

## Integration Guide

The specification's [System Overview](../spec/01_overview.md) provides a
high-level overview of the KMDB system for developers seeking to understand
how KMDB works, and the rest of the specification provides a deeper dive for
those interested in the inner workings of KMDB.

The "Integration Guide" will document how a developer can utilise KMDB in their
Dart or Flutter application. The document will describe how to create/open a
database; manage collections, documents and the vault; handle faults; etc. The
goal is to help developers to integrate KMDB into their application both quickly
and reliably.

As part of this guide, use a sample Flutter to-do application that demonstrates
the concepts described in the guide. The to-do application will have multiple
projects, each with their own list of tasks. Each task will have a title,
description, priority (high/medium/low), status (backlog/in-progress/done), file
attachments and comments (updates). This will provide a basis for demonstrating
database mechanics (open/close, update, create collection etc), schema,
indexing/search, vault, sync.

**Dependency:** this work moved out of `0_09.md` into its own roadmap because it
is a significant body of code and documentation in its own right — a new
`kmdb_example_todo` package plus the guide prose, not a docs-only change like
its former roadmap siblings. It depended on the
[Specification review and editing](completed/0_09.md) work item as a
prerequisite — the guide cites spec section numbers throughout, and that
review is what validates those sections are correct, complete, and current
before the guide builds on them. **That prerequisite is now satisfied**: `0_09`
is complete, so this plan can proceed.

> ### ⚠️ New dependency (2026-07-19): the security model this guide documents is changing
>
> The [2026-07-18 release-readiness review](../reviews/release-readiness-review-2026-07-18.md)
> found that `docs/spec/31_encryption.md` documents a **passive-only** threat
> model and never claims synced data is authentic (finding **E-1**). The
> decision taken in response is to **close that gap before `0.1.0`** — see the
> [0.10.01 hardening track](0_10_01.md).
>
> This guide teaches developers how to set up sync and the vault, so it will
> necessarily make security claims. Those claims are **about to change**:
> sync artefacts become authenticated, sync gains a device-enrollment step, and
> §31's threat model is being rewritten. Writing the guide first would document
> a model that no longer holds by the time `0.1.0` ships.
>
> **This work is therefore blocked on the 0.10.01 track** — specifically on
> Phases 6 and 9 of
> [plan_0_10_01_sync_trust_boundary.md](../plans/plan_0_10_01_sync_trust_boundary.md)
> (sync authentication, and the spec rewrite). The sample to-do app's sync and
> vault chapters are the affected parts; the earlier chapters (open/close,
> collections, documents, schema, indexing) are unaffected and could proceed in
> parallel if that is useful.
