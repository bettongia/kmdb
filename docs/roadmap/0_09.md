# v0.09

## Specification review and editing

The specification documents have grown over time and need to be reviewed.

This work will review all documents in `doc/spec/` and bring sections together
so that the specification is cohesive and comprehensive. Another important
aspect to this work will be to validate the specification against the codebase
as it stands - it is important that any gaps between the specification and the
code are resolved by either updating the spec or raising the need to modify the
code to meet the expected functionality.

## Integration Guide

The [KMDB Primer](../primer.md) provides an overview of the KMDB system to
developers seeking to understand how KMDB works and the specification can
provide a deeper dive for those interested in the inner workings of KMDB.

The "Integration Guide" will document how a developer can utilise KMDB in their
Dart or Flutter application. The document will describe how to create/open a
database; manage collections, documents and the vault; handle faults; etc. The
goal is to help developers to integrate KMDB into their application both quickly
and reliably.

As part of this guide, use a sample Flutter to-do application that demonstrates
the concepts described in the guide. The to-do application will have multiple
projects, each with their own list of tasks. Each task will have a title,
description, priority (high/medium/low), status (backlog/in-progress/done), file
attachments and comments (updates). This will provide a basis for demonstrating
database mechanics (open/close, update, create collection etc), schema,
indexing/search, vault, sync.

## Release Checklist

Releasing KMDB will require a sequence of publishing steps that ensure that each
package is published in the correct order. We'll need a release checklist
template that can be followed for each release.

A `README.md` file will be added to `docs/releasing` that describes the release
process and provides the template checklist.

Each release will have a copy of the template created in `docs/releasing` and
named after the version being released - e.g. `0.1.0-dev.1.md`. The version
provided in the main [`pubspec.yaml`](../../pubspec.yaml) will be the designated
version being released. It may be that some of the KMDB packages do not need to
be updated in a release so it is reasonable that different packages may have
different minor or patch versions but they must never have different major
versions.

The [Release Checklist specification](../spec/28_release_checklist.md) needs to
be included for each release as a pre-requisite series of tests that must be
completed prior to a release. Each item in the Release Checklist specification
must have an associated check-list entry that indicates if the test has been
completed (`[x]`) or if it is unnecessary for the release (`[-]`).

## Housekeeping

- [ ] **Remove stale `GatedSyncAdapter` duplicate.** `packages/kmdb/test/support/gated_sync_adapter.dart`
  is a dead copy of `packages/kmdb/lib/src/test_support/gated_sync_adapter.dart`. Update
  `sync_cancellation_integration_test.dart` to import from the `lib/` path and delete the
  duplicate. (Identified during coverage remediation — PR #49.)

- [ ] **`--recovery-code` success-path subprocess test.** The wrong-code → exit 1 case is
  covered in `cli_runner_inprocess_test.dart`, but no subprocess test verifies that a
  *correct* recovery code actually opens an encrypted DB and exits 0. Add to the
  `cli_runner_test.dart` subprocess suite. (Group A gap noted in coverage remediation — PR #49.)

- [ ] **`WalWriter.append` does not `syncDir` a fresh WAL file's directory entry.**
  `WalWriter.append`/`appendBatch` (`packages/kmdb/lib/src/engine/wal/wal_writer.dart`)
  `syncFile` the WAL content but never `syncDir` the WAL directory (`db-dir`). The flush
  path `syncDir`s only `sst/` (`lsm_engine.dart:862`), so a brand-new `wal-*.log` file's
  directory entry becomes durable only via an incidental later `syncDir(db-dir)` — at
  `open()` (device-identity write), on Manifest rotation, or during crash recovery. Latent,
  not currently live: today's paths do not depend on a fresh WAL surviving power loss before
  that incidental `syncDir`. Harden `WalWriter` to `syncDir` its directory on first write to
  a newly-created file (or have the rotate/flush path do so), so the guarantee is intrinsic
  rather than incidental. Invariant documented in §07 ("Directory-entry durability").
  (Identified during Phase 2 QA of `plan_0_08_encryption_confidentiality_reconciliation`,
  2026-07-11; the encryption plan added a defensive `syncDir` at its own call site as
  defence-in-depth, not a fix for this underlying gap.)

- [ ] **Intermittent flake in `sync_engine_test.dart` "H4-FU3 ingest-side floor still
  prevents resurrection".** `packages/kmdb/test/sync/sync_engine_test.dart:1170` passes
  20/20 in isolation but intermittently fails under full-suite parallel load with
  `Expected null, Actual [99]` (a deleted key appears to resurrect). **Diagnosis: a
  test-design flake, not a confirmed production bug.** Device B's "stale" pre-delete
  SSTable HLC in the test setup is wall-clock-derived and can land right at device A's
  ingest-floor boundary, making tombstone-vs-value admission scheduling-dependent when
  the suite runs under load. The suggested fix is **test-only** — give device B a
  deterministically sub-floor HLC in the test setup so admission is no longer timing-
  dependent — *not* a rewrite of the production ingest-floor logic, which appears
  correct. Needs a sync-literate follow-up to confirm the diagnosis (rule out a genuine
  resurrection path in the ingest floor) before applying the test-only fix; it should not
  be miscategorised as a data-loss defect on the basis of the symptom alone. (Identified
  during Phase 3 QA of `plan_0_08_encryption_confidentiality_reconciliation`, 2026-07-13;
  unrelated to and out of scope for the encryption plan.)

- [ ] **No test exercises a corrupted *encrypted* SQ8 vector at query time.** `VecManager`'s
  fixed-length corruption guard (`entry.value.length != expectedByteLen`) was moved to run
  post-decryption during the encryption confidentiality reconciliation plan, and is unit-
  tested against `EncryptionEnvelope` directly, but no test seeds a genuinely-corrupted
  *encrypted* SQ8 entry into an index and asserts it is skipped during an actual encrypted
  search query — the full integration path (encrypted-index query → corrupted entry →
  skip, not crash) is untested end-to-end. Add such a test to
  `packages/kmdb/test/search/semantic/vec_manager_test.dart` or
  `vault_searcher_test.dart`. (Identified during Phase 1 QA of
  `plan_0_08_encryption_confidentiality_reconciliation`, 2026-07-11; a completeness gap,
  not a known correctness risk — unit-level coverage of the primitive is strong.)

- [ ] **`MetaStore.appendTombstoneFloorAdvance`/`getTombstoneFloor` encryption
  read/write asymmetry.** `getTombstoneFloor` was updated to decrypt (`EncryptionEnvelope.
  unwrap`) as part of the encryption confidentiality reconciliation plan, but
  `appendTombstoneFloorAdvance` (the `WriteBatch`-based writer) still writes raw, unencrypted
  bytes — deliberately, since it has zero production call sites today (`LsmEngine.
  _compactAll` calls the standalone `setTombstoneFloor`, which *does* encrypt, directly) and
  wiring encryption into unused code was judged premature. A doc comment on
  `appendTombstoneFloorAdvance` (`packages/kmdb/lib/src/engine/kvstore/meta_store.dart`)
  warns of the asymmetry, but there is no tracked follow-up to actually fix it if the method
  ever gains a real caller — anyone wiring it up must remember to add encryption at that
  point, or a future encrypted-database read of a batch-written tombstone floor will
  silently misparse. (Identified during Phase 2 QA of
  `plan_0_08_encryption_confidentiality_reconciliation`, 2026-07-11.)

- [ ] **`EncryptionProvider.indexToken`'s domain separator (`":"`-join) has a theoretical
  concatenation ambiguity.** FTS/index HMAC token domains are built as `"{ns}:{field}:" +
  term` / `"{ns}:{path}:" + value`; the components are not escaped against an embedded
  literal `:`, so (in principle) a collection namespace or FTS term containing a literal
  colon could collide with a different `(ns, field, term)` triple that produces the same
  concatenated string. Vault-FTS (fixed 64-hex `sha256` domain) and secondary-index
  (hex-encoded final component) domains are structurally immune; only the FTS
  `ns`/`field`/raw-term domain is theoretically exposed. Documented as an acknowledged,
  deliberately-unfixed limitation in `EncryptionProvider.indexToken`'s doc comment
  (`packages/kmdb/lib/src/encryption/encryption_provider.dart`) rather than fixed, since
  KMDB namespace names are developer-controlled (not user input) and FTS terms are
  post-tokenisation (unlikely to contain a raw `:`) — low practical risk, but worth a proper
  escaping scheme (e.g. length-prefixing each component) if this primitive is ever reused
  for a domain where either constraint doesn't hold. (Identified during Phase 4 QA of
  `plan_0_08_encryption_confidentiality_reconciliation`, 2026-07-13.)
