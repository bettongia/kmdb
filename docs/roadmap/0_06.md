## WI-0: Local-only namespace segregation

**Prerequisite ‚Äî must land before WI-3 and before any new derived-index work.**

KMDB's sync protocol operates at the SSTable level: whole files are uploaded
verbatim. There is currently no mechanism to prevent device-local derived data ‚Äî
secondary indexes (`$index:*`), FTS term indexes (`$fts:*`), and embedding
vectors (`$vec:*`) ‚Äî from riding in those uploads. On a mobile connection with a
non-trivial database, syncing megabytes of derived index data that the receiving
device will rebuild anyway wastes user data significantly. Vector entries alone
are 384 bytes each; a large vault blob can produce hundreds of chunks.

¬ß12 and ¬ß20.7 both document this as a known gap ("Future work ‚Äî if upload-time
filtering is added‚Ä¶"). This WI builds that filter.

**Design:** a `$$` (double-dollar) namespace prefix marks any namespace as
local-only. The predicate `namespace.startsWith(r'$$')` is the single source of
truth. At flush time, the memtable is partitioned into two SSTables ‚Äî one
syncable, one local-only ‚Äî with a `localOnly` flag in the Manifest. The sync
engine skips local-only files at upload. Compaction preserves the partition.

Existing derived namespaces are renamed: `$fts:` ‚Üí `$$fts:`, `$vec:` ‚Üí `$$vec:`,
`$index:` ‚Üí `$$index:`. WI-3's new vault-specific namespaces follow the
`$$vault:` sub-namespace convention: `$vfts:` ‚Üí `$$vault:fts:`,
`$vault:extract:` ‚Üí `$$vault:extract:`. Vault chunk vectors have no KV namespace
at all ‚Äî stored only on the filesystem.

See
[Technical Proposal: Local-Only Namespace Segregation](../proposals/implemented/local_only_namespaces.md).

- **Scope:** core `kmdb` (LsmEngine, Manifest, SyncEngine, FtsManager,
  VecManager, IndexManager)
- **Dependencies:** none
- **Coordinates with:** v0.08 Gap 2 (HMAC token rename also touches FtsManager /
  IndexManager ‚Äî do in one pass where they overlap)
- **Plan:**
  [plan_0_06_wi0_local_only_namespaces.md](../plans/completed/plan_0_06_wi0_local_only_namespaces.md)
  ‚Äî status: Complete

## WI-1: Model identity & index invalidation

**Prerequisite ‚Äî must land before any embedding model change.**

`$$vec:` namespaces are local to each device and excluded from sync (once WI-0
is complete; see WI-0). Each device independently builds its vector index from
whatever model it has downloaded. When the model changes (e.g. BGE Small En ‚Üí a
multilingual model), the existing `$$vec:` index was built with different
embedding geometry and queries against it will return nonsense ‚Äî the index must
be invalidated and rebuilt.

Record the active embedding model ID in `$meta` on first write. On `open()` (and
after a model download), compare the stored model version against the active
model; if they differ, invalidate `$$vec:` and trigger a full index rebuild.

- **Scope:** core `kmdb` (MetaStore, VecManager)
- **Dependencies:** WI-0 (for accurate `$$vec:` namespace reference)
- **Plan:** ‚Äî (pre-implemented against old `$vec:` name; namespace rename
  delivered by WI-0)

## WI-2: Charset detection

`PlainTextExtractor` currently decodes `text/plain` bytes as UTF-8 and fails
silently on other encodings. Many plain-text files in the wild are ISO-8859-\*,
Windows-1252, Shift-JIS, or other legacy encodings ‚Äî incorrect decoding produces
garbled text that propagates into both the inverted index and embedding input,
corrupting both search paths.

`betto_charset_detector` is pure-Dart, published, and ready to use. Wire it into
`PlainTextExtractor.extract()` before UTF-8 decoding. The detected charset is
stored in `extract_status.json` for diagnostics (used by WI-3).

- **Scope:** core `kmdb`
- **Dependencies:** none
- **Available:** `betto_charset_detector` (published)
- **Plan:**
  [plan_0_06_wi2_charset_detection.md](../plans/completed/plan_0_06_wi2_charset_detection.md)

## WI-4: Multilingual embedding model

Adopt a multilingual embedding model to replace BGE Small En v1.5, delivering
broad-language semantic search with a single shared vector space. A multilingual
model places all languages in one coordinate system, enabling cross-lingual
retrieval without query-side language detection.

`multilingual-e5-small` is the recommended starting point: 384 dimensions
(drop-in ‚Äî no index format change), ~100 languages, MIT licence. `bge-m3` is the
upgrade path if long-document context or 1024-dim retrieval is later required
(full re-index needed). `bge-m3-v1.0` is already registered in
`betto_inferencing`'s `ModelCatalog` but not yet validated.

Key work in `betto_inferencing`:

- Write a pure-Dart XLM-R SentencePiece / Unigram tokenizer alongside the
  existing `BertTokenizer`. Port from `transformers.js` (TypeScript,
  Apache-2.0); use `tokenizers` (Rust) and `spm_precompiled` as byte-exact
  reference. Components: vocab loading, precompiled-charsmap normaliser,
  Metaspace pre-tokenizer, Unigram Viterbi, fairseq id remapping, `<s>‚Ä¶</s>`
  post-processing.
- Gate the tokenizer in CI against byte-exact token ID parity with HuggingFace
  `AutoTokenizer` on a fixed multilingual corpus.
- Validate the chosen model in `ModelCatalog` and apply E5 `passage:` / `query:`
  prefixes in the pipeline.

WI-1's model identity check triggers the index rebuild when the model changes on
existing databases.

**Investigation (2026-07-07) refined this WI's scope:** `bge-m3-v1.0`'s catalog
entry has placeholder (all-zero) checksums and is not a real registration ‚Äî
this WI targets `multilingual-e5-small` only, per the roadmap's own framing of
BGE-M3 as a later upgrade path, and **removes** the broken `bge-m3-v1.0` stub
entirely rather than leaving it in place (greenfields project, no cost to
re-adding it properly whenever BGE-M3 work is actually picked up). `OnnxEmbeddingModel`/`BertTokenizer` are
hard-wired together (no shared tokenizer abstraction exists yet) and
`EmbeddingModel.embed()` has no passage/query distinction ‚Äî both are addressed
by the plan below (a new `ModelTokenizer` abstraction and an `EmbeddingKind`
parameter). Whether to hand-port the SentencePiece/Unigram tokenizer from
`transformers.js` (as originally assumed) or adopt the pub.dev package
`dart_sentencepiece_tokenizer` is resolved by a parity spike in the plan's
Phase 0, not decided in advance. `kmdb` core, the vault search path, and the
CLI are already model-agnostic ‚Äî the bulk of this WI's work is in
`betto_inferencing`.

- **Scope:** `betto_inferencing`, core `kmdb` (pipeline wiring)
- **Dependencies:** WI-1
- **Plan:**
  [plan_0_06_wi4_multilingual_embedding_model.md](../plans/plan_0_06_wi4_multilingual_embedding_model.md)

**Status (2026-07-09):** Phases 0‚Äì2 complete and merged into
`betto_inferencing` (published as `0.1.0-dev.3` ‚Äî `EmbeddingKind`,
`ModelTokenizer`/`XlmRobertaTokenizer`, `multilingual-e5-small` registered and

sign-off status.

## WI-5: Language detection (`betto_lang_detector`)

New pure-Dart package `betto_lang_detector`. Language detection serves a narrow
role ‚Äî lexical analyzer selection, per-document metadata, and cheap script-level
routing for the BM25 tokenizer (WI-6). The semantic path does not need it
because the multilingual model (WI-4) shares one vector space across all
languages.

A two-stage pure-Dart detector is sufficient:

1. **Unicode script pre-filter** ‚Äî resolves or sharply narrows many languages
   from script alone (Han, Cyrillic, Arabic, Devanagari, Thai, ‚Ä¶). Near-free, no
   model required.
2. **Small character n-gram model** ‚Äî distinguishes same-script languages (e.g.
   English / German / French in Latin script). Constrained to known languages
   via `restrictTo` for accuracy.

Key API: `LanguageDetector` (with `detect()` and `dominantScript()`),
`DetectionResult` (`Detected` / `Undetermined`), `restrictTo` parameter.

Supersedes the earlier options considered (MediaPipe tflite, FastText/Floret,
Lingua-rs) ‚Äî all were heavier or required non-Dart runtimes. A pure-Dart
approach has no native dependencies and runs in the indexing isolate on all
platforms.

- **Scope:** new `betto_lang_detector` package
- **Dependencies:** WI-2 (charset detection is the Phase A prerequisite)
- **Available:** `betto_lang_detector` 0.1.0-dev.1, published on
  [pub.dev](https://pub.dev/packages/betto_lang_detector); source at
  [bettongia/lang_detector](https://github.com/bettongia/lang_detector)
- **Plan:**
  [plan_0_06_wi5_lang_detector.md](../plans/completed/plan_0_06_wi5_lang_detector.md)
  _(status: Complete ‚Äî implemented in the standalone repo, not in this
  workspace)_

## WI-6: Language-aware BM25 tokenizer routing

**Revised scope, discovered during investigation.** The original framing above
(route non-Latin-script content from a default `RegExpTokenizer` to
`IcuTokenizer` via `dominantScript()`) turned out to be stale: document-field
FTS (`FtsManager`, ¬ß21) was **already** calling `createDefaultTokenizer()`
(‚Üí `IcuTokenizer` on native, `BrowserTokenizer` on web) at every read/write
site, independent of language ‚Äî there was no live tokenizer bug to route away
from there, and no routing work was needed.

The real, live bug was narrower and in a different place: `VaultChunker` (the
vault search chunker introduced in WI-3) used its own hand-rolled ASCII-only
`RegExp(r"\w+(?:'\w+)*")` ‚Äî no Unicode flag ‚Äî which matched **zero** tokens
for text with no ASCII letters/digits, silently producing an unsearchable
(but still `indexed`) blob for pure-CJK/Arabic/Cyrillic/Devanagari/Thai vault
content. What actually shipped:

1. **`VaultChunker` tokenizer fix** ‚Äî a new `OffsetTokenizer` interface
   (`TokenSpan`, added upstream in `betto_icu`/`betto_lexical`) lets the
   chunker use the same `IcuTokenizer` the vault query path already used,
   instead of the broken regex, while still recovering the character offsets
   the chunker needs for byte-span bookkeeping.
2. **Script/language metadata** ‚Äî `VaultExtractionState` gained `script`
   (ISO 15924, from `dominantScript()`) and `language` (ISO 639-1, from
   detection) as two separate fields (not a single combined value), stored in
   the `$$vault:extract:{sha256}` KV entry (`VaultExtractionState`) ‚Äî **not**
   a filesystem `extract_status.json` file, which WI-10's investigation
   confirmed does not exist (see ¬ß32).
3. **Language-aware stemming, both search paths** ‚Äî extended
   `betto_lexical`'s `Stemmer` (previously English-only) to all 28 languages
   the underlying `snowball_stemmer` package implements, then wired both
   document-field FTS (`FtsManager`) and vault search (`VaultChunker` /
   `VaultSearcher`) to select a stemmer via a shared
   `detectLanguageForStemming()` helper. Naive confidence-threshold gating
   (even down to `0.0`) proved unreliable for short/keyword-style text ‚Äî
   `LanguageGuess.confidence` degenerates to `1.0` on a spuriously-won tie ‚Äî
   so the shipped policy instead gates on the margin between the top two
   ranked candidates, a minimum word count, and Stemmer-language support,
   defaulting to English otherwise (see ¬ß21 Stage 4 for the full policy and
   its empirical justification).

Tokenizer *routing* between `IcuTokenizer` and `RegExpTokenizer` ‚Äî the
original framing ‚Äî was never built, because it was never needed.

- **Scope:** core `kmdb` (`VaultChunker`, vault indexing isolate,
  `VaultSearcher`, `FtsManager`, `pipeline.dart`); upstream prerequisites in
  `betto_icu` (`OffsetTokenizer`) and `betto_lexical` (`Stemmer` language
  extension + `OffsetTokenizer` re-export), both published as `0.1.0-dev.2`
- **Dependencies:** WI-5
- **Available:** `betto_lang_detector` (WI-5), `betto_icu`/`betto_lexical`
  `0.1.0-dev.2`
- **Plan:**
  [plan_0_06_wi6_bm25_tokenizer_routing.md](../plans/completed/plan_0_06_wi6_bm25_tokenizer_routing.md)
  _(status: Complete)_

## WI-7: Magika second-pass (opt-in)

Add an optional Magika ONNX inference step inside the vault indexing isolate
(introduced in WI-3) to validate whether a blob flagged as `text/plain` is
actually readable text. Surfaces the `is_text` flag and a confidence score;
stores `detectorScore` in the `$$vault:extract:{sha256}` KV entry (there is no
filesystem `extract_status.json` file ‚Äî see the WI-10 investigation and ¬ß32).
`betto_mediatype_detector` remains the canonical detector at ingest time ‚Äî
Magika does not modify `manifest.json`.

**Magika is opt-in via `VaultSearchConfig` (default off).** This keeps the
default resource footprint unchanged, which matters on mobile. The isolate loads
and disposes the Magika session before loading the embedding model ‚Äî the two ORT
sessions are never resident simultaneously.

A working Magika ONNX example already exists in `betto_onnxrt`; the model itself
is a few MB (significantly lighter than the embedding model).

- **Scope:** vault indexing isolate (native-only), `VaultSearchConfig` API
- **Dependencies:** WI-3
- **Available:** Magika example in `betto_onnxrt` (needs surfacing as a callable
  API, not just a CLI demo)

## WI-8: PDF extractor (`kmdb_extractor_pdf`)

**Complete.** New optional package `kmdb_extractor_pdf` implements the
`VaultTextExtractor` interface (introduced in WI-3) for `application/pdf`
content via `PdfTextExtractor`, wrapping
[`betto_pdfium`](https://pub.dev/packages/betto_pdfium) 0.1.0-dev.3 (PDFium
FFI). No core `kmdb` changes were needed ‚Äî the extension point shipped with
WI-3/WI-10. `PdfTextExtractor` extracts via `PdfDocument.extractPlainText()`
and gates predominantly-scanned/image-only documents behind a configurable
`scannedPageRatio` (default 0.5): above the threshold, `extract()` returns
`""` rather than indexing sparse OCR fragments, mapping to the existing
`indexed`/0-chunks status with no core enum change. Multi-column and RTL
extraction quality are **not** documented guarantees of `betto_pdfium` ‚Äî v1
ships with multi-column test coverage but no verified RTL support; see the
plan's Q3 resolution for what fixture coverage exists.

Vault search indexing is native-only (¬ß20/CLAUDE.md), so `betto_pdfium`'s
web/WASM support is unused by this package; scope is macOS, Linux, Windows,
iOS, Android, with iOS requiring the companion `betto_pdfium_ios` Flutter
plugin in the consuming app (mirrors `betto_onnxrt`/`betto_onnxrt_ios`).

Two pre-existing, unrelated `package:kmdb` core bugs were discovered during
implementation and left as out-of-scope follow-up work (see the plan's
Summary): a `$vault` ref-count key-length mismatch that blocks writing a
vault-URI-referencing document via the public collection API, and a
`VaultStore.listFilesRecursive` v1 stopgap that silently hides all blobs from
`vaultIndexingStatus()`/recovery/GC unless overridden.

- **Scope:** new `kmdb_extractor_pdf` package
- **Dependencies:** WI-3
- **Available:** `betto_pdfium` 0.1.0-dev.3 (published; all platforms)
- **Plan:**
  [plan_0_06_wi8_pdf_extractor.md](../plans/completed/plan_0_06_wi8_pdf_extractor.md)

## WI-10: Encrypt `extract/` filesystem artifacts

**Complete.** When database encryption is enabled (¬ß31), vault blob bytes are
encrypted at rest by `VaultStore`. The derived filesystem artifacts written by
the vault search indexer were previously written as plaintext. `text.txt` in
particular was a significant information leak: it contains the full extracted
plaintext of a blob whose ciphertext is otherwise opaque.

The investigation (see the plan) found the shipped WI-3 implementation never
writes a fourth `extract_status.json` file ‚Äî status lives only in the
`$$vault:extract:{sha256}` KV entry ‚Äî narrowing this WI's scope to the three
artifacts that actually exist on disk: `text.txt`, `chunks_v1.json`, and
`vectors_*.bin`.

The three files are now encrypted using the database DEK (AES-256-GCM,
consistent with ¬ß31 value encryption) whenever an `EncryptionProvider` is
configured. When encryption is not configured, files remain plaintext,
preserving pre-WI-10 behaviour byte-for-byte.

Key design points, as implemented:

- Encryption/decryption of `extract/` files is handled by
  `VaultSearchManager.writeExtractArtifact` / `readExtractArtifact`, using the
  `EncryptionProvider` passed at construction time.
- The indexing isolate produces plaintext artifacts; the main isolate encrypts
  them before writing to disk (keeping the DEK off the isolate boundary,
  consistent with the blob-bytes pattern in WI-3).
- Each file is prefixed with a single self-describing `EncryptionFlag` byte
  (the same enum ¬ß31 defines for the `ValueCodec` wire format) rather than
  gating on a global "database is encrypted" check, because `extract/` files
  have no accompanying manifest and encryption can be toggled on for a
  database that already has plaintext artifacts. This makes every file
  independently readable regardless of the database's current encryption
  state or when it was written.
- If encryption is toggled on an existing database, `extract/` artifacts
  written before the change remain plaintext (flag `0x00`) until the affected
  blobs are reindexed (`VaultSearchManager.reindexVault()`), at which point
  they are rewritten with the encrypted flag (`0x01`). Both flag states
  coexist correctly and indefinitely across blobs in the same database. See
  ¬ß31 and ¬ß32 for the full spec.

- **Scope:** core `kmdb` (vault search manager, vault searcher, filesystem
  write/read paths)
- **Dependencies:** WI-3 ‚Äî implemented immediately after WI-3 and before any
  additional extractors (WI-7, WI-8, WI-9), so the encryption foundation is in
  place before the extractor ecosystem expands.
- **Plan:**
  [plan_0_06_wi10_encrypt_extract_artifacts.md](../plans/completed/plan_0_06_wi10_encrypt_extract_artifacts.md)

## WI-11: XLM-R SentencePiece tokenizer (lands in `betto_inferencing`)

**Discovered during WI-4's Phase 0 investigation spike (2026-07-07), not
originally planned.** WI-4 targets `multilingual-e5-small`, an XLM-RoBERTa-family
model requiring SentencePiece/Unigram subword tokenization. The spike evaluated
adopting the pub.dev package `dart_sentencepiece_tokenizer` rather than
hand-porting a tokenizer from `transformers.js` as WI-4's own text originally
assumed. Result: **0/11 smoke-corpus entries matched byte-exact** ‚Äî root cause
confirmed at the source level: the package parses HuggingFace's `Precompiled`
charsmap normalizer bytes (`tokenizer.json`'s `normalizer.normalizers[]`, a
base64 `precompiled_charsmap` field) but never applies them ‚Äî `_buildNormalizerSpec`
in `huggingface_json.dart` reads the whitespace/dummy-prefix flags correctly but
drops the charsmap bytes entirely, always effectively normalizing as identity for
that step. Per WI-4's Q6, this triggered a hard gate: no tokenizer-port code was
written in `betto_inferencing`; a dedicated design pass is required first.

Follow-up investigation (during this WI's own planning) found the gap is
narrower than "rebuild a tokenizer": vocabulary loading, the Metaspace-equivalent
whitespace/prefix/escape handling, Unigram Viterbi decoding, and BOS/EOS
post-processing (`SentencePieceConfig.addBosToken`/`addEosToken`) all work
correctly via `dart_sentencepiece_tokenizer`'s existing public API. The missing
piece is specifically the charsmap-trie lookup algorithm (SentencePiece's "Darts"
double-array trie format) ‚Äî this doesn't exist in any known Dart package and
must be built. Because the library's `SentencePieceTokenizer` constructor is
private, a corrected normalizer can't be injected by subclassing ‚Äî but
pre-applying a correct `normalize()` to raw text before calling the library's
public `encode()` composes cleanly (matches SentencePiece's own internal pass
ordering: charsmap substitution, then whitespace collapse). No fork/vendor of
the third-party package is expected to be needed for the currently understood
scope; if the dependency's public API changes incompatibly in the future, its
MIT licence permits vendoring a pinned copy as a fallback (compatible with this
project's Apache-2.0 licensing).

review pass):** lands as new files directly inside `betto_inferencing`
