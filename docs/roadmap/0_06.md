## WI-0: Local-only namespace segregation

**Prerequisite — must land before WI-3 and before any new derived-index work.**

KMDB's sync protocol operates at the SSTable level: whole files are uploaded
verbatim. There is currently no mechanism to prevent device-local derived data —
secondary indexes (`$index:*`), FTS term indexes (`$fts:*`), and embedding
vectors (`$vec:*`) — from riding in those uploads. On a mobile connection with a
non-trivial database, syncing megabytes of derived index data that the receiving
device will rebuild anyway wastes user data significantly. Vector entries alone
are 384 bytes each; a large vault blob can produce hundreds of chunks.

§12 and §20.7 both document this as a known gap ("Future work — if upload-time
filtering is added…"). This WI builds that filter.

**Design:** a `$$` (double-dollar) namespace prefix marks any namespace as
local-only. The predicate `namespace.startsWith(r'$$')` is the single source of
truth. At flush time, the memtable is partitioned into two SSTables — one
syncable, one local-only — with a `localOnly` flag in the Manifest. The sync
engine skips local-only files at upload. Compaction preserves the partition.

Existing derived namespaces are renamed: `$fts:` → `$$fts:`, `$vec:` → `$$vec:`,
`$index:` → `$$index:`. WI-3's new vault-specific namespaces follow the
`$$vault:` sub-namespace convention: `$vfts:` → `$$vault:fts:`,
`$vault:extract:` → `$$vault:extract:`. Vault chunk vectors have no KV namespace
at all — stored only on the filesystem.

See
[Technical Proposal: Local-Only Namespace Segregation](../proposals/implemented/local_only_namespaces.md).

- **Scope:** core `kmdb` (LsmEngine, Manifest, SyncEngine, FtsManager,
  VecManager, IndexManager)
- **Dependencies:** none
- **Coordinates with:** v0.08 Gap 2 (HMAC token rename also touches FtsManager /
  IndexManager — do in one pass where they overlap)
- **Plan:**
  [plan_0_06_wi0_local_only_namespaces.md](../plans/completed/plan_0_06_wi0_local_only_namespaces.md)
  — status: Complete

## WI-1: Model identity & index invalidation

**Prerequisite — must land before any embedding model change.**

`$$vec:` namespaces are local to each device and excluded from sync (once WI-0
is complete; see WI-0). Each device independently builds its vector index from
whatever model it has downloaded. When the model changes (e.g. BGE Small En → a
multilingual model), the existing `$$vec:` index was built with different
embedding geometry and queries against it will return nonsense — the index must
be invalidated and rebuilt.

Record the active embedding model ID in `$meta` on first write. On `open()` (and
after a model download), compare the stored model version against the active
model; if they differ, invalidate `$$vec:` and trigger a full index rebuild.

- **Scope:** core `kmdb` (MetaStore, VecManager)
- **Dependencies:** WI-0 (for accurate `$$vec:` namespace reference)
- **Plan:** — (pre-implemented against old `$vec:` name; namespace rename
  delivered by WI-0)

## WI-2: Charset detection

`PlainTextExtractor` currently decodes `text/plain` bytes as UTF-8 and fails
silently on other encodings. Many plain-text files in the wild are ISO-8859-\*,
Windows-1252, Shift-JIS, or other legacy encodings — incorrect decoding produces
garbled text that propagates into both the inverted index and embedding input,
corrupting both search paths.

`betto_charset_detector` is pure-Dart, published, and ready to use. Wire it into
`PlainTextExtractor.extract()` before UTF-8 decoding. The detected charset is
stored in `extract_status.json` for diagnostics (used by WI-3).

- **Scope:** core `kmdb`
- **Dependencies:** none
- **Available:** `betto_charset_detector` (published)
- **Plan:**
  [plan_0_06_wi2_charset_detection.md](../plans/completed/plan_0_06_wi2_charset_detection.md)

## WI-3: Vault search — core

The main v0.06 feature. Implements the architecture in
[Technical Proposal: Vault File Search](../proposals/vault_search.md) for
`text/plain` content:

- `VaultTextExtractor` interface + `PlainTextExtractor` (uses WI-2 for charset
  decoding)
- Chunking pipeline (`chunks_v1.json`, `text.txt`, byte offsets)
- Storage namespaces: `$$vault:fts:`, `$$vault:extract:`, `$vault:docref:`;
  chunk vectors stored only in `extract/vectors_{modelId}_sq8.bin` (not in the
  LSM); `extract/` directory layout per §4 of the proposal
- Async indexing isolate with startup recovery (§5 of the proposal)
- `KmdbCollection.searchVault()` API (lexical / semantic / hybrid modes)
- `extract_status.json` lifecycle state machine

The plan must resolve the open questions in §8 of the proposal (unified vs.
separate search API, re-index trigger, indexing observability surface, stub blob
hydration trigger, LSM scan index rebuild path).

from the plan review (2026-06-21). It requires revision to incorporate the `$$`
namespace convention, the `VaultSearchResult<T>` standalone type, and other
review resolutions before it can reach `Investigated` status.

- **Scope:** core `kmdb`
- **Dependencies:** WI-0, WI-1, WI-2

## WI-4: Multilingual embedding model

Adopt a multilingual embedding model to replace BGE Small En v1.5, delivering
broad-language semantic search with a single shared vector space. A multilingual
model places all languages in one coordinate system, enabling cross-lingual
retrieval without query-side language detection.

`multilingual-e5-small` is the recommended starting point: 384 dimensions
(drop-in — no index format change), ~100 languages, MIT licence. `bge-m3` is the
upgrade path if long-document context or 1024-dim retrieval is later required
(full re-index needed). `bge-m3-v1.0` is already registered in
`betto_inferencing`'s `ModelCatalog` but not yet validated.

Key work in `betto_inferencing`:

- Write a pure-Dart XLM-R SentencePiece / Unigram tokenizer alongside the
  existing `BertTokenizer`. Port from `transformers.js` (TypeScript,
  Apache-2.0); use `tokenizers` (Rust) and `spm_precompiled` as byte-exact
  reference. Components: vocab loading, precompiled-charsmap normaliser,
  Metaspace pre-tokenizer, Unigram Viterbi, fairseq id remapping, `<s>…</s>`
  post-processing.
- Gate the tokenizer in CI against byte-exact token ID parity with HuggingFace
  `AutoTokenizer` on a fixed multilingual corpus.
- Validate the chosen model in `ModelCatalog` and apply E5 `passage:` / `query:`
  prefixes in the pipeline.

WI-1's model identity check triggers the index rebuild when the model changes on
existing databases.

- **Scope:** `betto_inferencing`, core `kmdb` (pipeline wiring)
- **Dependencies:** WI-1
- **Plan:** —

## WI-5: Language detection (`betto_lang_detector`)

New pure-Dart package `betto_lang_detector`. Language detection serves a narrow
role — lexical analyzer selection, per-document metadata, and cheap script-level
routing for the BM25 tokenizer (WI-6). The semantic path does not need it
because the multilingual model (WI-4) shares one vector space across all
languages.

A two-stage pure-Dart detector is sufficient:

1. **Unicode script pre-filter** — resolves or sharply narrows many languages
   from script alone (Han, Cyrillic, Arabic, Devanagari, Thai, …). Near-free, no
   model required.
2. **Small character n-gram model** — distinguishes same-script languages (e.g.
   English / German / French in Latin script). Constrained to known languages
   via `restrictTo` for accuracy.

Key API: `LanguageDetector` (with `detect()` and `dominantScript()`),
`DetectionResult` (`Detected` / `Undetermined`), `restrictTo` parameter.

Supersedes the earlier options considered (MediaPipe tflite, FastText/Floret,
Lingua-rs) — all were heavier or required non-Dart runtimes. A pure-Dart
approach has no native dependencies and runs in the indexing isolate on all
platforms.

- **Scope:** new `betto_lang_detector` package
- **Dependencies:** WI-2 (charset detection is the Phase A prerequisite)
- **Available:** `betto_lang_detector` 0.1.0-dev.1, published on
  [pub.dev](https://pub.dev/packages/betto_lang_detector); source at
  [bettongia/lang_detector](https://github.com/bettongia/lang_detector)
- **Plan:**
  [plan_0_06_wi5_lang_detector.md](../plans/completed/plan_0_06_wi5_lang_detector.md)
  _(status: Complete — implemented in the standalone repo, not in this
  workspace)_

## WI-6: Language-aware BM25 tokenizer routing

Use `dominantScript()` from `betto_lang_detector` to route non-Latin-script
content to `IcuTokenizer` (via `betto_icu` / `betto_lexical`, already available)
in place of the default `RegExpTokenizer`. This fixes CJK, Arabic, Cyrillic, and
other scripts where space-splitting produces incorrect or empty token sequences.

Applies to both the existing document-field FTS (`FtsManager`, §21) and vault
FTS (the indexing isolate introduced in WI-3). The `"language"` field is
stored in the `$$vault:extract:{sha256}` KV entry (`VaultExtractionState`) —
**not** a filesystem `extract_status.json` file, which WI-10's investigation
confirmed does not exist (see §32).

- **Scope:** core `kmdb` (FtsManager, vault indexing isolate)
- **Dependencies:** WI-5
- **Available:** `IcuTokenizer` in `betto_icu` / `betto_lexical` (already a
  dependency)
- **Plan:** —

## WI-7: Magika second-pass (opt-in)

Add an optional Magika ONNX inference step inside the vault indexing isolate
(introduced in WI-3) to validate whether a blob flagged as `text/plain` is
actually readable text. Surfaces the `is_text` flag and a confidence score;
stores `detectorScore` in the `$$vault:extract:{sha256}` KV entry (there is no
filesystem `extract_status.json` file — see the WI-10 investigation and §32).
`betto_mediatype_detector` remains the canonical detector at ingest time —
Magika does not modify `manifest.json`.

**Magika is opt-in via `VaultSearchConfig` (default off).** This keeps the
default resource footprint unchanged, which matters on mobile. The isolate loads
and disposes the Magika session before loading the embedding model — the two ORT
sessions are never resident simultaneously.

A working Magika ONNX example already exists in `betto_onnxrt`; the model itself
is a few MB (significantly lighter than the embedding model).

- **Scope:** vault indexing isolate (native-only), `VaultSearchConfig` API
- **Dependencies:** WI-3
- **Available:** Magika example in `betto_onnxrt` (needs surfacing as a callable
  API, not just a CLI demo)
- **Plan:** —

## WI-8: PDF extractor (`kmdb_extractor_pdf`)

**Complete.** New optional package `kmdb_extractor_pdf` implements the
`VaultTextExtractor` interface (introduced in WI-3) for `application/pdf`
content via `PdfTextExtractor`, wrapping
[`betto_pdfium`](https://pub.dev/packages/betto_pdfium) 0.1.0-dev.3 (PDFium
FFI). No core `kmdb` changes were needed — the extension point shipped with
WI-3/WI-10. `PdfTextExtractor` extracts via `PdfDocument.extractPlainText()`
and gates predominantly-scanned/image-only documents behind a configurable
`scannedPageRatio` (default 0.5): above the threshold, `extract()` returns
`""` rather than indexing sparse OCR fragments, mapping to the existing
`indexed`/0-chunks status with no core enum change. Multi-column and RTL
extraction quality are **not** documented guarantees of `betto_pdfium` — v1
ships with multi-column test coverage but no verified RTL support; see the
plan's Q3 resolution for what fixture coverage exists.

Vault search indexing is native-only (§20/CLAUDE.md), so `betto_pdfium`'s
web/WASM support is unused by this package; scope is macOS, Linux, Windows,
iOS, Android, with iOS requiring the companion `betto_pdfium_ios` Flutter
plugin in the consuming app (mirrors `betto_onnxrt`/`betto_onnxrt_ios`).

Two pre-existing, unrelated `package:kmdb` core bugs were discovered during
implementation and left as out-of-scope follow-up work (see the plan's
Summary): a `$vault` ref-count key-length mismatch that blocks writing a
vault-URI-referencing document via the public collection API, and a
`VaultStore.listFilesRecursive` v1 stopgap that silently hides all blobs from
`vaultIndexingStatus()`/recovery/GC unless overridden.

- **Scope:** new `kmdb_extractor_pdf` package
- **Dependencies:** WI-3
- **Available:** `betto_pdfium` 0.1.0-dev.3 (published; all platforms)
- **Plan:**
  [plan_0_06_wi8_pdf_extractor.md](../plans/completed/plan_0_06_wi8_pdf_extractor.md)
  _(status: Complete)_

## WI-10: Encrypt `extract/` filesystem artifacts

**Complete.** When database encryption is enabled (§31), vault blob bytes are
encrypted at rest by `VaultStore`. The derived filesystem artifacts written by
the vault search indexer were previously written as plaintext. `text.txt` in
particular was a significant information leak: it contains the full extracted
plaintext of a blob whose ciphertext is otherwise opaque.

The investigation (see the plan) found the shipped WI-3 implementation never
writes a fourth `extract_status.json` file — status lives only in the
`$$vault:extract:{sha256}` KV entry — narrowing this WI's scope to the three
artifacts that actually exist on disk: `text.txt`, `chunks_v1.json`, and
`vectors_*.bin`.

The three files are now encrypted using the database DEK (AES-256-GCM,
consistent with §31 value encryption) whenever an `EncryptionProvider` is
configured. When encryption is not configured, files remain plaintext,
preserving pre-WI-10 behaviour byte-for-byte.

Key design points, as implemented:

- Encryption/decryption of `extract/` files is handled by
  `VaultSearchManager.writeExtractArtifact` / `readExtractArtifact`, using the
  `EncryptionProvider` passed at construction time.
- The indexing isolate produces plaintext artifacts; the main isolate encrypts
  them before writing to disk (keeping the DEK off the isolate boundary,
  consistent with the blob-bytes pattern in WI-3).
- Each file is prefixed with a single self-describing `EncryptionFlag` byte
  (the same enum §31 defines for the `ValueCodec` wire format) rather than
  gating on a global "database is encrypted" check, because `extract/` files
  have no accompanying manifest and encryption can be toggled on for a
  database that already has plaintext artifacts. This makes every file
  independently readable regardless of the database's current encryption
  state or when it was written.
- If encryption is toggled on an existing database, `extract/` artifacts
  written before the change remain plaintext (flag `0x00`) until the affected
  blobs are reindexed (`VaultSearchManager.reindexVault()`), at which point
  they are rewritten with the encrypted flag (`0x01`). Both flag states
  coexist correctly and indefinitely across blobs in the same database. See
  §31 and §32 for the full spec.

- **Scope:** core `kmdb` (vault search manager, vault searcher, filesystem
  write/read paths)
- **Dependencies:** WI-3 — implemented immediately after WI-3 and before any
  additional extractors (WI-7, WI-8, WI-9), so the encryption foundation is in
  place before the extractor ecosystem expands.
- **Plan:**
  [plan_0_06_wi10_encrypt_extract_artifacts.md](../plans/completed/plan_0_06_wi10_encrypt_extract_artifacts.md)
