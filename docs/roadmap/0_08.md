# v0.08

## Encryption confidentiality reconciliation

**Status: Complete.** Plan:
[plan_0_08_encryption_confidentiality_reconciliation.md](../plans/completed/plan_0_08_encryption_confidentiality_reconciliation.md).
All four gaps below are resolved — see §31 (_Known gaps and unprotected
surfaces_) for the current, authoritative confidentiality-boundary
statement. Landed as five phase commits on one branch/PR (Phase 0 was
design-only, no commit): Phase 1 (Gap 1, value encryption), Phase 2 (Gap 3,
`MetaStore`), Phase 3 (Gap 4, vault manifest `originalName`), Phase 4
(Gap 2, HMAC namespace tokens), Phase 5 (this doc/spec/glossary update).
This is a **breaking on-disk format change with no migration path** —
databases created before this work item landed must be recreated (§31's
_Database Format-Version Gate_ section, `LegacyDatabaseFormatException`,
`docs/spec/28_release_checklist.md` RC-22).

An audit conducted during v0.06 planning (validated by the `kmdb-architect`
agent, 2026-06-21) identified a cluster of gaps between §31's stated
confidentiality guarantees and the actual implementation. This work item closed
the most impactful ones before the v1 beta.

### Gap 1 — FTS and Vec values are unencrypted (code/spec contradiction) — resolved

§31 lines 273–277 explicitly state: _"System namespace values (indexes, FTS,
vector, versioning) are encrypted when encryption is active … Encrypting index
entries ensures that cloud storage never sees plaintext document content in any
form."_

This was false as implemented (historical description below; current
namespace names are `$$`-prefixed per WI-0 — see §31 gap 1, now marked
resolved). `FtsManager` and `VecManager` wrote index values
using raw `cbor.encode()`, not `ValueCodec.encode(encryption:)`. As a result:

- `$$fts:doc:{ns}:{field}` values leaked the full tokenised term list for
  each document.
- `$$vec:{ns}:{field}` values leaked raw SQ8-quantised embedding vectors.
- Both `$$fts:`/`$$vec:` namespaces are local-only and never uploaded to
  cloud storage (WI-0 correction — the plaintext exposure was to local
  SSTable access, not cloud storage; see §31 gap 1 for the resolved,
  accurate framing).

**Fix:** wire `EncryptionProvider` through all `FtsManager` and `VecManager`
value write and read paths, consistent with how `KmdbCollection`,
`IndexManager`, and `VersionManager` already do it. This is a targeted,
mechanical change — no storage format or schema change is required, since the
`ValueCodec` wire format already has the `EncryptionFlag` byte.

The same audit anticipated that the vault-search entries introduced by WI-3
(v0.06) — renamed by WI-0 to `$$vault:fts:`, `$$vault:vec:`, and
`$$vault:extract:` — would have the same problem unless written through
`ValueCodec.encode(encryption:)` from the start, and flagged it for
verification once WI-3 shipped.

**Confirmed during WI-10 planning (2026-07-02): not implemented.** WI-3 shipped
with the same defect. `VaultBm25Writer` and `VaultVecWriter` write
`$$vault:fts:`/`$$vault:vec:` values via raw `cbor.encode()`
(`vault_bm25_writer.dart:174-182`, doc comment: _"...these are internal index
entries that never need compression or encryption at this layer..."_), and
`VaultExtractionState.encode()` (`vault_extraction_state.dart:236`) does the
same for `$$vault:extract:` (doc comment: _"Uses bare CBOR (no `ValueCodec`
wrapper) consistent with the other vault [namespaces]"_). All three vault
namespaces are therefore in scope for this Gap 1 fix alongside `$$fts:`/
`$$vec:`. Note WI-10 (encrypt `extract/` filesystem artifacts) is a distinct,
narrower work item — it covers `text.txt`/`chunks_v1.json`/`vectors_*.bin` on
disk, not these KV values — and does not fix this gap.

### Gap 2 — FTS and secondary index namespace names leak terms and field values — resolved for encrypted databases

**Resolved** by Phase 4, largely as recommended below — see §31 gaps 2/3 and
the §99 glossary "Index token" entry for the final, implemented description.
The historical namespace forms referenced in this write-up predate WI-0's
`$$`-prefix rename and WI-3's vault-search namespace naming; the current
(and implemented) forms are `$$fts:{ns}:{field}:{token}`,
`$$index:{ns}:{path}:{token}`, and `$$vault:fts:{sha256}:{token}`, where
`{token}` is a plaintext hex encoding on an unencrypted database or an
HMAC-SHA256 token on an encrypted one (`EncryptionProvider.indexToken`).
`VecIndexState` was found during implementation to need no equivalent
migration tracking — `$$vec:`/`$$vault:vec:idx:` namespaces are keyed by
document ID/chunk index, never an embedded term or value, so there was no
hex-tokenised scheme to migrate away from (see the plan's Phase 4
checklist).

The namespace-per-term layout (`$fts:{ns}:{field}:{hexTerm}`,
`$index:{ns}:{field}:{value}`, and the vault-search equivalent
`$vfts:{sha256}:{hexTerm}`) embeds the hex-encoded term or indexed value in the
namespace name. Namespace names are part of the SSTable key and are never
encrypted. An attacker with local SSTable access could enumerate the
full search vocabulary and indexed field values by scanning namespace names,
even after Gap 1 is fixed (these namespaces are local-only, never uploaded
to cloud storage — see the Gap 1 correction above).

**Recommended fix: DEK-keyed HMAC token namespace names** (from the
`kmdb-researcher` investigation, 2026-06-21). Replace `hexTerm`/`hexValue` with
`HMAC-SHA256(indexSubKey, domain ‖ term)[0:16].hex`, where `indexSubKey` is an
HKDF sub-key derived from the DEK (`info = "kmdb-index-token"`). At query time,
reconstruct the namespace name the same way using the in-memory DEK — no engine
or `KvStore` changes are required, since the engine treats namespace names as
opaque byte strings.

Key properties confirmed by the investigation:

- The internal SSTable key format (`[nsLen][ns UTF-8][userKey 16B][hlc 8B]`)
  treats the namespace as an opaque, length-prefixed byte string. The engine's
  `scan()` does exact-namespace prefix matching — replacing hex bytes with HMAC
  bytes is structurally invisible to the storage engine, compaction, Bloom
  filters, and sync protocol.
- Domain-separate the HMAC over `"{ns}:{field}:" + term` (FTS) /
  `"{ns}:{path}:" + value` (index) so the same term in different fields maps to
  different tokens.
- When encryption is off, fall back to today's plaintext hex (tokens are only
  meaningful with a DEK).
- Index/FTS state must record whether built in HMAC mode; toggling encryption
  triggers a rebuild rather than silently querying the wrong namespaces.
- **DEK rotation:** §31's passphrase rotation re-wraps the DEK but never changes
  it, so HMAC tokens survive rotation. A future "change the DEK" feature would
  require a full index rebuild — document this constraint.

**Residual leakage (accepted, not closed):** the HMAC scheme closes direct
vocabulary enumeration. Statistical side-channels inherent to any searchable
symmetric encryption scheme — term frequency, search-pattern leakage,
co-occurrence, document count per term — remain observable. These require
ORAM-grade machinery to close and are disproportionate for KMDB's threat model
(untrusted cloud _storage_, not an active query-observing adversary). They are
documented as accepted limitations in §31's "Threat Model & Confidentiality
Boundaries" section.

**Sequencing note for WI-3 (vault search, v0.06) — outcome:** WI-3 shipped
(as `$$vault:fts:`, post-WI-0 rename) well before this plan's Phase 4, so
"day one HMAC adoption" did not apply in practice; instead Phase 4 added a
per-blob `ftsTokenMode` migration marker on `VaultExtractionState` (mirroring
`FtsIndexState`/`IndexState`'s equivalent) so any blob indexed by pre-Gap-2
code is detected and migrated (stale hex namespaces purged, blob
re-extracted under HMAC tokens) the next time `VaultSearchManager.recover()`
runs — see §31 gap 2's "Format-version migration" note.

**Alternative layout rejected:** a single `$fts:{ns}:{field}` namespace with
`{hexTerm}:{docId}` as the key was evaluated and rejected — `scan()` is
exact-namespace-only with no cross-namespace prefix API, and the engine's
16-byte user-key decoding would require engine-level changes. The current
namespace-per-term layout is load-bearing; key the namespace name instead.

**Relevant source files:**

- `src/engine/util/key_codec.dart` — `encodeInternalKey` (opaque namespace
  format)
- `src/engine/util/namespace_codec.dart` — 255-byte cap, single-source-of-truth
  pattern to mirror
- `src/engine/kvstore/lsm_engine.dart` lines 489–558 — `scan()` exact-namespace
  matching
- `src/search/lexical/fts_manager.dart` — Gap A: raw `cbor.encode` at lines
  1073/1091/1109/1125/1252; Gap B: `_termNamespace`/`_termToHex` at lines
  999–1008
- `src/query/index/index_writer.dart` — `_encodeValueHex`,
  `indexNamespaceForValue`

**Open questions for the plan — resolved (see the plan's Q3/Q4/Q5):**

- HMAC key: DEK directly, or HKDF sub-key (`info = "kmdb-index-token"`)?
  **Resolved: HKDF sub-key**, exactly as recommended — implemented as
  `EncryptionProvider.indexToken`.
- Empty `$index:` value: encrypt anyway (29-byte overhead, uniformity) or leave
  empty (HMAC namespace already provides confidentiality)? **Resolved: left
  empty** — the namespace token alone carries the (now-opaque) value.
- Migration: rebuild indexes on next open, or require explicit `db.reindex()`
  call? **Resolved: automatic rebuild on next open/recover()**, gated on a
  `tokenMode`/`ftsTokenMode` format-version discriminator, mirroring WI-1's
  model-identity invalidation — not a runtime encryption toggle (impossible;
  see §31's _Database Format-Version Gate_ and _Provisioning Guard_).

### Gap 3 — `MetaStore` values bypass `ValueCodec` — resolved

`MetaStore` wrote raw CBOR directly to `_engine.put()`, bypassing
`ValueCodec.encode(encryption:)` entirely. The `enc:blob` exemption remains
intentional and documented (§31 bootstrap requirement). The remaining `$meta`
entries — namespace registry (which leaks collection names), device ID, HLC
timestamps, generation counters, and model identity — were unencrypted.

**Resolved** by Phase 2: every general `$meta` accessor now routes through
`EncryptionEnvelope` when a provider is configured. The `enc:blob` entry
remains unencrypted (bootstrap invariant), enforced structurally —
`getEncryptionBlob`/`putEncryptionBlob` call the engine directly, bypassing
the newly-encrypting general accessors entirely. This introduced a
breaking on-disk format change for **every** existing database (not just
encrypted ones) — see §31's _Database Format-Version Gate_ and
`docs/spec/28_release_checklist.md` RC-22.

### Gap 4 — Vault `manifest.json` leaks blob metadata — `originalName` resolved

`manifest.json` is plaintext JSON on disk and in cloud sync, containing
`schemaVersion`, `mediaType`, `size`, `createdAt`, `crc32c`, `sha256`, and
`originalName` (field names as of the current `VaultManifest`; this
write-up's `hlcTimestamp` was a stale name for `createdAt`).

- `sha256` is intentionally computed over plaintext bytes (§31, dedup
  guarantee). Acceptable; document explicitly.
- `originalName` (the user-supplied filename) was not acknowledged as a
  plaintext surface in §31. For an encrypted database, this leaked the
  original filename.
- `mediaType`, `size`, `crc32c`, and `createdAt` reveal content type,
  approximate content volume, and secondary provenance metadata.

**Resolved for `originalName`** by Phase 3: `VaultStore.ingest` encrypts it
in place (`EncryptionEnvelope` + base64) before it is written into
`manifest.json`, keeping the manifest's JSON shape stable; `VaultStore.
getManifest` is the sole decryption point. `mediaType`, `size`, `crc32c`,
and `createdAt` remain **intentionally plaintext**, each for a stated
functional reason (sync routing, dedup, extractor selection) — see §31 gap
5 for the accepted-limitation reasoning per field, not addressed by this
plan.

### Sequencing (as executed)

Phase 1 (Gap 1, FTS/Vec/vault-search value encryption) landed first, as the
highest-priority and most mechanical fix. Phase 2 (Gap 3, `MetaStore`) came
next. Phase 3 (Gap 4, manifest `originalName`) followed. Phase 4 (Gap 2,
namespace-token leakage) landed last — the largest and riskiest phase,
deliberately sequenced after the other three so its `EncryptionProvider`/
`MetaStore` dependencies (the provider-binding order established by Phase
2, and the format-version-gate precedent it set) were already in place.
Phase 5 (this doc/spec/glossary update) closed out the plan.

- **Scope:** core `kmdb` (`FtsManager`, `VecManager`, `MetaStore`,
  `VaultManifest`, `IndexWriter`/`IndexManager`, `VaultSearchManager`)
- **Dependencies:** none (independent of v0.06 vault search work)
- **Plan:**
  [plan_0_08_encryption_confidentiality_reconciliation.md](../plans/completed/plan_0_08_encryption_confidentiality_reconciliation.md)

## Vault file export

The CLI should include a `vault export` command that will export a specific
vault file to the path given in the `--output` parameter.

- If `--output` is a file path, the vault file will be exported to that path.
- If `--output` is a directory, the vault file will be placed into the requested
  directory and the file will be named based on the `originalName` property in
  `metadata.json` - if the `originalName` property does not exist, the vault
  file with be exported to the path with a file name of `blob`.

Note that the CLI usage indicates that the user can run `vault help` but this
fails on a database where Vault has not been initialised
(`Error: Vault is not available for this database. Vault storage is initialised automatically when files are first ingested via "vault ingest" or "--import".`).
This needs to be fixed as part of this work if it hasn't already been rectified.

- **Scope:** `kmdb_cli` (`vault_command.dart`, new `vault_export_command.dart`,
  `repl/completer.dart`); doc fix in `docs/spec/24_vault.md`
- **Dependencies:** none
- **Plan:** [plan_0_08_vault_file_export.md](../plans/plan_0_08_vault_file_export.md)

## `kmdb_cli` cloud sync credentials: OS keychain storage

The Encryption confidentiality reconciliation work item (above) deliberately
scoped Google Drive OAuth credential storage out of its own remit (Q7):
`remote_config.dart` stores `AccessCredentials.toJson()` as plaintext JSON
under the per-machine `local/` directory. This is not a database-encryption
gap — the credentials never sync, are never written into an SSTable, and
have no relationship to the DEK/`enc:blob` even in principle — but it is a
genuine local-secret-at-rest exposure: anyone with filesystem access to the
CLI's `local/` directory can read a live OAuth token and act as the user
against their cloud storage, independent of whether the KMDB database
itself is encrypted. See §31 gap 9 for the accepted-limitation framing this
work item is meant to close.

**Fix:** store CLI-managed cloud credentials in the OS-native credential
store instead of a plaintext file — macOS Keychain, Windows Credential
Manager (DPAPI), and the Secret Service API (libsecret) on Linux, with a
plaintext-file fallback only where no OS keychain is available (e.g. a
headless/CI environment) and an explicit warning when that fallback is
used. This is CLI-only; it does not touch `kmdb` core, the encryption
provider, or any synced/database-encrypted surface.

Open questions for the plan: which Dart/Flutter keychain-access package (if
any) covers all three desktop platforms without a native-asset build hook
mismatch with the rest of the workspace; whether other CLI-managed secrets
(if any accrue over time) should share the same storage mechanism; and
what the fallback UX should look like on headless/CI hosts where no OS
keychain is reachable.

- **Scope:** `kmdb_cli` (`remote_config.dart`, and any other CLI-managed
  credential storage found during investigation)
- **Dependencies:** none — independent of `kmdb` core and of the Encryption
  confidentiality reconciliation work item, which already shipped
- **Plan:** not yet written
