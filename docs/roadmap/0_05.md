# v0.05

> Review [KMDB Roadmap Review — 2026-06-05](../reviews/roadmap-review-2026-06-05.md)
> before planning any work in this release.

v0.05 makes KMDB's native-binary dependencies work across all target platforms
and establishes the CI signing pipeline. It is a **hard prerequisite** for any
feature work in `kmdb_inferencing` on mobile, and for web sync correctness
(WASM decompression, see §betto_zstd below).

Two packages require cross-platform binary support:

| Package | Mechanism | Note |
|---|---|---|
| `betto_zstd` | `native_toolchain_c` + WASM | Web decompression non-deferrable |
| `kmdb_inferencing` → `betto_onnxrt` | ONNX Runtime FFI | iOS/Android unblocked by extraction |

`betto_registry` (used by `MediaTypeDetector`) is pure Dart — no binary work
required. `betto_icu` extraction is complete.

### [x] betto_zstd — WASM and multi-platform pipeline

Plan: [betto_zstd pipeline](../plans/plan_betto_zstd_pipeline.md) —

- [x] WASM compression + decompression (Emscripten self-built WASM; frame
  compatibility guaranteed by construction — same C source as native path)
- [x] Frame-compat verification: golden-file test (`test/frame_compat_test.dart`)
  covers native→WASM, WASM→native, and WASM round-trip
- [x] Multi-platform CI (macOS, Linux x86_64, Android, iOS, Windows x64/MSVC,
  Web/Chrome)
- [x] `VERSION_ZSTD` file + build-hook assertion (parses `#define` from `zstd.h`)
- [x] pub.dev publishing (`publish_to: none` removed)

Q6 (vendored C source update automation) deferred to a follow-up.

Key constraints:

- [`zstandard`](https://pub.dev/packages/zstandard) is the candidate WASM
  back-end; frame compatibility must be verified empirically before adoption.
- Apple Archive does not support Zstd; `native_toolchain_c` is the approach
  for iOS/macOS.
- Android: [Zstd-jni](https://github.com/luben/zstd-jni) bundles pre-compiled
  `.so` for all major ABIs; added to the **host app's** `build.gradle`.

### [ ] betto_onnxrt extraction and iOS/Android support

Plan: [Extract betto_onnxrt](../plans/completed/plan_betto_onnxrt_extraction.md) —
Proposal: [betto_onnxrt design](../proposals/betto_onnxrt.md)

Stages A and B complete (PR #42, 2026-06-10). iOS SPM shim and Android
host-app wiring remain open.

- [x] Stage A: generic `betto_onnxrt` package scaffold + `ModelDownloader`
  (`github.com/bettongia/onnxrt`, CI green on macOS/Linux/Windows)
- [x] Stage B: replace in-tree `kmdb_inferencing` types; `UnsupportedError`
  documented for iOS pending SPM shim
- [ ] iOS: bundle ORT XCFramework via SPM (`onnxruntime-c`); wired into
  `kmdb_ui` or consumer app — plan: [`plan_ios_spm_shim.md`](https://github.com/bettongia/onnxrt/blob/main/plans/plan_ios_spm_shim.md)
- [ ] Android: Maven Central AAR (`onnxruntime-android`) wired into **host
  app** (`kmdb_ui`); `betto_onnxrt` native-assets path removes this later
- [x] `VERSION_ONNX` file + `tool/generate_versions.dart` → `versions.g.dart`
- [x] Multi-platform CI (macOS, Linux x86_64, Windows x64)

Key constraints:

- **iOS:** Use Swift Package Manager (`onnxruntime-c` XCFramework). Do
  **not** use CocoaPods (deprecated). Do **not** use `onnxruntime-mobile` —
  reduced opset breaks the existing BGE model.
- **Semantic search is opt-in on iOS** — ORT framework adds ~10–15 MB
  post-thinning; model downloads on first use.
- **ONNX models are data assets, not executables** — downloaded and cached
  on first use on all platforms, including iOS (App Store permits data
  downloads). Vector indexes are local-only, excluded from sync.

## GitHub Actions signing pipeline

- [ ] **macOS**: Developer ID certificate + `notarytool` notarization for
  `betto_zstd.dylib` and `kmdb_cli`. Store certificate and App Store Connect
  API key as Actions secrets. ORT dylib (from Microsoft) is pre-signed.
- [ ] **Windows**: Authenticode OV certificate. ⚠ EV certificates are
  hardware-bound and impractical in CI — investigate cloud HSM signing as an
  alternative before treating Windows as unblocked.
- **iOS / Android**: Signing handled by Xcode / Flutter toolchain; no
  additional pipeline work for native libraries.

## Future: Magika (v0.06 candidate)

[Magika](https://github.com/google/magika) (Google file-type detection) is a
candidate for vault search in v0.06. Approach options: Rust FFI bindings, or
the [ONNX model](https://github.com/google/magika/tree/main/assets/models/standard_v3_3)
(~3 MB; pin to a specific release, e.g.
[rust-v0.2.1](https://github.com/google/magika/tree/rust-v0.2.1/assets/models/standard_v3_3)).
No v0.05 work required.
