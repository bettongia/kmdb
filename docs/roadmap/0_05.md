## Multi-platform pipelines

Presently a number of packages that are used by KMDB rely on platform-specific
binaries:

- `kmdb_inferencing`: uses the ONNX runtime
- `betto_zstd`: provides the Zstandard library

The following architectures need to be supported:

- macOS: Universal Binary (combining Intel x86_64 and Apple Silicon arm64) using
  lipo (where possible)
- iOS
- Linux: x86_64, arm64, and aarch64 (ARMv8 64-bit - for Raspberry Pi)
  - Use Podman/Docker
- Android
- Windows:
  - Use `MinGW-w64` in Podman container
- Web
  - WASM

Ensuring build provenance is key. Developers should be able to locally build the
required binaries - especially when testing new versions. Production releases
require the binaries to be built by a GitHub pipeline (actions).

Use of any binary will be pinned to a specific version via a `VERSION_*` file at
the package root — for example `VERSION_ONNX` containing `v1.22.0` and
`VERSION_ZSTD` containing `v1.5.7`. These files are the **single source of
truth**: CI scripts, local helper scripts (e.g. `scripts/download_ort.sh`), and
the Dart runtime all derive the version from them.

To prevent drift, version constants in Dart source (e.g. `_ortVersion` in
`ort_library.dart`) are **generated** from the VERSION files rather than
hardcoded. A `tool/generate_versions.dart` script reads each VERSION file and
writes a `lib/src/generated/versions.g.dart` file containing the relevant
constants. This script runs as part of the CI pipeline and should be run locally
via `make generate` before building or testing.

### GitHub Actions pipeline

Production binaries must be built and signed by the GitHub Actions pipeline, not
by individual developers. The following signing requirements apply:

- **macOS**: Native libraries (e.g. `betto_zstd.dylib`) must be signed with a
  Developer ID certificate and notarized via `notarytool` before distribution.
  The `kmdb_cli` executable requires the same treatment if distributed as a
  standalone binary. Developer ID certificate and App Store Connect API key must
  be stored as Actions secrets. The ONNX Runtime dylib downloaded from GitHub is
  already signed by Microsoft and requires no further action.
- **Windows**: Binaries should be Authenticode-signed. A standard OV
  (Organisation Validated) certificate is suitable for CI use. Note that EV
  (Extended Validation) certificates are hardware-bound and impractical in a CI
  environment — treat this as a potential blocker and investigate CI-compatible
  EV alternatives (e.g. cloud HSM signing services) if EV is required.
- **iOS / Android**: Signing is handled by the Xcode / Flutter build toolchain
  and the respective app store submission pipelines — no additional pipeline
  work required for native libraries.

### ONNX

The [ONNX runtime](https://onnxruntime.ai/) and binaries are available via the
[GitHub project](https://github.com/microsoft/onnxruntime/tags).

`scripts/download_ort.sh` is used to download the required runtime for local
development.

The [onnxruntime_v2](https://pub.dev/packages/onnxruntime_v2) is an option but
`kmdb_inferencing` already contains the required bindings.

Once ONNX Runtime is available for all platforms, the various models used by
KMDB will operate correctly (no need for platform-specific `.onnx` files).

Note that ONNX Runtime is available in GPU and non-GPU distributions.

#### Models

ONNX models (`.onnx` files) are **data assets, not executable code**, and are
therefore downloaded and cached on first use rather than bundled with the
application. This applies on all platforms, including iOS, where App Store rules
prohibit downloading executable code at runtime but explicitly permit
downloading data.

Vector indexes (`$vec:` namespaces) are **local to each device** and excluded
from sync. This is intentional: models may differ across devices and index
rebuilds are inexpensive relative to sync complexity. A device that has not yet
downloaded a model simply has no vector index until first use.

Model identity tracking (recording which model version built a given index, so
that index invalidation can be triggered on model change) was completed in
[plan_configurable_embedding_model.md](../plans/completed/plan_configurable_embedding_model.md)
(PR #39).

**LFS asset removal (deferred — follow-up plan required).** The bundled BGE
model data files under `packages/kmdb_inferencing/assets/models/` are still
checked into Git LFS (deferred as Q5 in the completed configurable embedding
model plan). Now that `ModelDownloader` is in place, these should be removed.
Prerequisites before filing the plan:

1. `betto_onnxrt` Stage A wired in and download path proven in CI.
2. A CI strategy chosen for the inferencing test suite (most likely: a small
   purpose-built fixture ONNX model checked in as a plain binary, not LFS, that
   satisfies the BGE input/output shape without real weights — enabling full test
   coverage without a 127 MB CI download).

File a plan under `docs/plans/` once the above prerequisites are met.

#### iOS

The ONNX Runtime framework must be **bundled at build time** — iOS does not
permit downloading executable code at runtime. Use the
[Swift Package Manager repo](https://github.com/microsoft/onnxruntime-swift-package-manager)
(`onnxruntime-c` XCFramework). Do **not** use CocoaPods (being deprecated) and
do **not** use the `onnxruntime-mobile` variant: it uses a reduced opset and
requires a custom model build, breaking compatibility with the existing BGE
model.

Semantic search is **opt-in** on iOS. The ONNX Runtime framework adds roughly
10–15 MB (post-App-Store thinning, single architecture); the model is an
additional download on first use. If bundle size proves a shipping blocker, the
`mobile` variant with a model-specific custom build can be revisited.

`ort_library.dart` currently throws `UnsupportedError` for `Platform.isIOS` and
needs an iOS branch that resolves the framework-bundled library path.

#### Android

Use the official Maven Central AAR:

```gradle
implementation 'com.microsoft.onnxruntime:onnxruntime-android:<version>'
```

This bundles `.so` files for all standard ABIs (arm64-v8a, armeabi-v7a, x86,
x86_64) and places them on the JNI path automatically. The existing
`DynamicLibrary.open('libonnxruntime.so')` call in `ort_library.dart` works
unchanged. Constrain to `arm64-v8a` via `ndk { abiFilters 'arm64-v8a' }` if
binary size is a concern.

This Gradle dependency goes in the **host application** (`kmdb_ui`), not in the
`kmdb_inferencing` package itself — document this for all consumer apps.

The Dart native-assets / build-hook mechanism is the planned long-term path and
will be delivered via the `betto_onnxrt` package (see §"betto_onnxrt" below),
which removes the Gradle dependency from the host app entirely.

### ZStandard

The approach is to build and ship all binaries from source, pinned to a specific
version via a `VERSION_ZSTD` file. This maximises resilience against third-party
sources disappearing and ensures build provenance.

A vendored pure-Dart fallback (along the lines of `just_zstd`) is kept as a
contingency only — it is not the primary path.

#### Web (WASM)

The web platform cannot use the native FFI path. Real Zstd compression on web is
important because OPFS storage is constrained — shrinking SSTables directly
extends how much data a web client can hold.

The approach is a **WebAssembly build of Zstd invoked via `dart:js_interop`**,
producing frames byte-compatible with native `betto_zstd`. Frame compatibility
is the gating requirement: a web-written SSTable must decompress on a native
client and vice versa, so this must be verified before the web Zstd path ships.

The [`zstandard`](https://pub.dev/packages/zstandard) pub.dev package (which
uses a WASM build on web) is a candidate, but must be confirmed to work in a
pure-Dart/non-Flutter web context before adoption.

#### Windows

Not available

#### Linux

Not available

#### iOS & MacOS

Apple provides the
[Apple Archive framework](https://developer.apple.com/documentation/applearchive/),
which DOES NOT natively support zstd compression.

#### Android

The easiest method is to use the official
[Zstd-jni GitHub repository](https://github.com/luben/zstd-jni), which bundles
pre-compiled native binaries for all major Android architectures (arm64-v8a,
armeabi-v7a, x86, x86_64) into a single dependency you can add to your
build.gradle.

### betto_onnxrt

The ORT binary distribution, FFI binding, and model-cache infrastructure planned
for `kmdb_inferencing` are candidates for extraction into a standalone
`betto_onnxrt` package (separate repo, following the `betto_zstd` convention).
See [docs/proposals/betto_onnxrt.md](../proposals/betto_onnxrt.md) for the full
design.

Plan:
[Extract `betto_onnxrt` — Standalone ONNX Runtime Package](../plans/plan_betto_onnxrt_extraction.md)

**Sequencing:** `plan_configurable_embedding_model.md` (PR #39, complete) placed
`ModelSpec`/`ModelDownloader` in `kmdb_inferencing` with BGE-shaped types.
Stage B of `plan_betto_onnxrt_extraction.md` replaces those in-tree types with
the generic `betto_onnxrt` equivalents. The iOS gap in `ort_library.dart` closes
as part of that extraction. This plan is a v0.05 prerequisite for any
`kmdb_inferencing` feature work on mobile platforms.

### Magika

Google's [Magika](https://github.com/google/magika) is slated for possible
future integration (see [0_06.md](0_06.md)).

The
[Magika website](https://securityresearch.google/magika/introduction/overview)
indicates that its core is built in Rust so FFI bindings could/should work.
Alternatively, an
[ONNX Model is available in the repo](https://github.com/google/magika/tree/main/assets/models/standard_v3_3) -
use of a specific release would be prudent (e.g.
[v0.2.1 of the Rust package](https://github.com/google/magika/tree/rust-v0.2.1/assets/models/standard_v3_3)).
The model file is small - about 3Mb.
