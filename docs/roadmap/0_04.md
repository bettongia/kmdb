# v0.04

## SAHPool OPFS

Plan: plans/plan_sahpool_opfs.md

Migrate `StorageAdapterWeb` from async File System API to Sync Access Handles in
a dedicated Web Worker for 3–4× throughput.

## Cloud Adapter: Google Drive

Plan: plans/plan_google_drive_sync.md

Directly connect to Google Drive for storing sync objects such as the SSTables
etc.

- Drive REST API + `If-Match` ETag
- `GcsAdapter` (GCS JSON API), implements `SyncStorageAdapter`
- Each adapter lives under `lib/src/sync/cloud/`.

## Flutter UI — Mobile (iOS and Android)

Plan: plans/plan_flutter_ui_mobile.md

Extend `kmdb_ui` to iOS and Android, validating the KMDB engine on mobile
platforms. This milestone pairs naturally with the Google Drive adapter: cloud
sync is what makes mobile sync viable.

Scope:

- **Build validation**: confirm `betto_zstd` (FFI, `native_toolchain_c`) and
  `kmdb_tokenizer_icu` link and load on iOS arm64 and Android arm64. This is
  the gating question — if Zstd fails on iOS, compression is disabled there
  (it is optional; the engine can store uncompressed values).
- **Database path UX**: mobile apps cannot use a directory file picker. Databases
  live under `getApplicationDocumentsDirectory()/<name>/` on both platforms.
  Vault storage follows the same root: `<dbDir>/vault/`. The desktop
  `FilePicker.getDirectoryPath()` flow is guarded behind a platform check.
- **Feature suppression**: sync hidden on mobile (filesystem sync is not
  practical; cloud sync deferred to this milestone's Google Drive adapter).
  Semantic search hidden on iOS (`kmdb_inferencing` / ONNX Runtime is
  unsupported there); available on Android.
- **Adaptive layout**: the navigator-push stack from the desktop plan is already
  wired; smoke-tested on a 390 px iPhone and 360 px Android screen.
