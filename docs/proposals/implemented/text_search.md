# Technical Proposal: Text Indexing and Search

## 1. Executive Summary

Whilst kmdb currently provides for the storage and indexing of fields this is
limited to storage of the "full value" rather than handling the richness of
language usually stored in textual information. Whilst some text (string) values
may suit this approach (e.g. enumerations such as 'open', 'in-progress' and
'complete'), further work is required to provide for full-text search.

Rudimentary filtering of text-based values via operators such as `eq`, `ne`,
`contains`, `startsWith` and `endsWith` lack the capabilities provided by
lexical (keyword) and semantic search. Lexical indexing and search has a long
history of development and implementations such as
[Apache Lucene](https://lucene.apache.org/) and
[Tantivy](https://github.com/quickwit-oss/tantivy). Adding the ability for a
user to create a lexical search index will provide kmdb with a more useful
search approach for textual values.

In more recent years semantic search, based in vector embeddings generated by
language models, has provided the ability to search semantically across a
corpus. Embedding models are used to generate a vector based on text and locate
similar items within the multi-dimensional space. The main concern for
local-first, multi-platform systems such as kmdb is that many solutions rely on
hosted LLMs to provide the generation of embeddings. However, work in this space
on embedding models has reached a point where the model can be hosted locally
(even on resource-limited devices). There are a number of available models (see
the [Ollama site](https://ollama.com/search?c=embedding) for examples) with a
couple of likely candidates for this work:

- [`all-MiniLM-L6-v2`](https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2)
  (~22MB, 384d, English) — smallest production-quality option
- [`multilingual-e5-small`](https://huggingface.co/intfloat/multilingual-e5-small)
  (~120MB, 384d) — if multi-language support is needed

Finally, a hybrid approach will provide the ability to combine the results of a
lexical search with those returned from a semantic search so as to provide the
user with the most appropriate results list

## 2. Sub-proposals

This proposal consists of 3 sub-proposals, each outlined in their own document
as per below (in priority order):

1. [Lexical search](lexical_search.md)
2. [Semantic search](semantic_search.md)
3. [Hybrid search](hybrid_search.md)

Sub-proposals 1 and 2 have no pre-requisite implementation and could be
undertaken in parallel. Sub-proposal 3 could also be undertaken in parallel but
will be deferred until the completion of 1 and 2.

The rest of the document outlines key constraints that apply to all 3 of the
underlying proposals.

## 3. Constraints/Requirements

### 3.1 English-language centric

The initial work described in this proposal (and the 3 sub-proposals) will only
focus on English language text. This covers text that would be associated with
the `en` language tag from the IETF/BCP 47 language codes.

However, design decisions made here must take future internationalisation
improvements into account. Examples include:

1. Ensuring "pluggable" approaches to elements such as tokenisation, stemming
   and embedding that allow for different algorithms/models to be selected based
   on the language of the source text. Different languages require different
   approaches to natural language processing in order to be effective and future
   work will be considered for broader internationalisation (i18n) and
   localisation (l10n) across kmdb.
1. Where one possible approach is similar to another, the one that supports the
   broadest set of languages must be chosen. For example, use of the Unicode
   Segmentation algorithm (UAX #29) would be preferred over algorithms with more
   limited language coverage/applicability.

#### Out of scope

The JSON format lacks a built-in model for encapsulating the language presented
in a value - unlike XML & HTML, which utilise the `lang` and `xml:lang`
properties. Future work on kmdb will need to determine approaches that allow for
the following:

1. Setting of a default (human) language for the whole database. This will most
   likely utilise the IETF/BCP 47 language codes.
2. Indication in documents regarding the use of specific languages at the
   document and value levels. Work in this space will investigate the use of
   approaches such as JSON-LD that offer concepts around
   [String internationalization](https://www.w3.org/TR/json-ld/#string-internationalization)

### 3.2 Document value-based indexing only

This proposal is limited to the indexing of a String value within a kmdb
document. It is expected that the string value is in UTF-8 text written in the
English (`en`) language - no check will be made by the system to ensure this.

#### Out of scope

The text indexing approach for file-based attachments will be determined in the
[vault](vault.md) proposal. Whilst it is expected that the same
processes/algorithms can be used as the ones described here, the size of the
attachments can present a significant difference from the size of a document
record in kmdb.

### 3.3 Optional enablement

The user must be in control of which indexing approach is used. It is likely
that creation of the lexical and semantic search indexes will be provided via
existing approaches (such as the `index` command in the CLI). This will allow
the user to create either or both type of search indexes as they deem necessary.

As index definition and generation is local, the user is also able to determine
if they wish to utilise a certain type of index based on their device. For
example, the user may wish to utilise a semantic index on their desktop device
but not mobile device. Whilst the technology (e.g. GPU or NPU) and storage
required is increasingly available on many devices the processing power for
generating vectors may see some users elect not to utilise the approach.

Naturally, the option to filter results through the hybrid approach requires
that both type of index has been configured.

### 3.4 Dart-centric

The implementation work must not rely on the Flutter SDK — it must be a truly
portable Dart implementation. This rules out any package that declares a Flutter
dependency.

### 3.5 Web browser implementation is out of scope

Adoption of this proposal on Web Browser-based deployments is to be deferred to
a later effort because its viability for both model inference and vector
indexing is unconfirmed.

Running LiteRT/WASM in a Web Worker for inference, plus IVF-Flat vector search
either on the main thread or in the same worker, has significant complexity. The
IVF-Flat row in the Phase C table should not be considered a committed approach
until this phase's investigation is complete.

Key questions to resolve in future work includes:

- Is the vector indexing even possible in a web environment, considering memory
  constraints?
- Can a model be compiled to WASM and run inside a Web Worker without blocking
  the main thread?
- What is the WASM binary size impact on initial page load?
- Is there a meaningful accuracy difference between running the embedding model
  in WASM vs. natively that would affect search quality?
- If WASM inference is not viable, is server-side embedding generation
  (offloading to a kmdb server) an acceptable fallback? (See future
  possibilities in §4.)
- Is it more useful to defer to a "server-based" deployment of kmdb that could
  provide model-based services for web deployments?

## 4. Additional out of scope items

The following are explicitly out of scope for this proposal. They may be
addressed in separate future proposals.

- **Autocomplete / search-as-you-type** — prefix-completion UI behaviour is not
  covered. The lexical index will support full-term keyword search only.
- **Plain text extraction** — all content submitted for indexing is assumed to
  arrive as plain text encoded in UTF-8. Extracting text from formats such as
  PDF, HTML, or DOCX is the responsibility of the caller.
- **Document summarisation and PDF processing** — using a language model to
  summarise or extract text from documents is out of scope.
- **Generative model use cases** — the language model included in this proposal
  is scoped strictly to producing text embeddings for semantic search. Agent,
  Q&A, and summarisation capabilities are not part of this work.

## 5. Future Possibilities

The following capabilities are out of scope for this proposal but may become
feasible once the embedding infrastructure from this work is in place. Each
would require a separate proposal.

- **Document summarisation** — using a generative model to summarise documents
  stored in the [vault](vault.md). Would require adding a generative model
  alongside (or instead of) the embedding model chosen in Phase A.
- **PDF and rich-text processing** — extracting plain text from PDF, HTML, and
  other formats prior to indexing. Out of scope here because this proposal
  assumes plain text input (§3).
- **Agent / Q&A** — an assistant that answers questions using stored documents
  as context. Requires a generative model; out of scope for this proposal.
- **Server-side index generation** — a server version of kmdb that generates and
  synchronises indexes centrally. This would help support the web implementation
  (offloading embedding work to the server) but introduces security and
  encryption requirements. kmdb must remain local-first; the web version would
  still function without a server, just with reduced search capabilities.
